# 大数据分析与处理工程师结构化面试标准

## 岗位信息

- 公司：云智数科有限公司
- 岗位编号：JOB-DATA-001
- 部门：数据平台研发部

## 1. 核心能力维度

- 批流处理：Spark、Flink、Kafka 的处理模型、状态和容错。
- 数据建模：离线数仓、实时数仓、数据湖与湖仓分层。
- 数据可靠性：幂等、去重、迟到乱序、质量规则和故障恢复。
- 性能优化：数据倾斜、大规模 SQL、分区、存储和资源治理。
- 数据治理：口径、元数据、血缘、权限和变更影响分析。

## 2. 结构化面试问题

1. Kafka 数据发生重复消费时，如何定位原因并保证下游结果幂等？
2. Flink Checkpoint 持续失败时，你会检查哪些指标、日志和依赖？
3. Spark Join 出现严重数据倾斜时，如何识别倾斜 Key 并选择优化方案？
4. 如何设计同时服务经营分析和 Agent 数据需求的离线与实时数仓？
5. 如何建立数据质量规则和端到端数据血缘，并处理口径变更？
6. 如何处理事件时间下的迟到数据和乱序数据？
7. 请说明一个你参与的数据任务的规模、延迟、资源和优化指标。

## 3. 优秀回答特征

- 区分消息重复投递、消费失败、Offset 提交时机和下游重复写入，使用业务键、幂等写入、去重状态和补偿机制。
- 能从 Checkpoint 大小、耗时、超时、存储可用性、反压、网络和状态后端逐层定位。
- 先通过任务指标和分区分布确认倾斜，再比较预聚合、广播 Join、加盐、拆分热点和分区调整。
- 数仓设计包含源数据、明细、汇总、应用层及实时链路，并说明口径、主键、历史拉链和回溯策略。
- 能定义完整性、唯一性、一致性、及时性和准确性规则，并将血缘用于变更通知与影响评估。
- 结合 Watermark、允许迟到、状态 TTL、侧输出和补算说明业务取舍。

## 4. 一般回答特征

- 知道 Kafka Offset、Flink Checkpoint 和 Spark 分区的概念，但缺少生产排查顺序。
- 能画出基础数仓分层，但不能解释实时与离线口径一致性或历史数据修正。
- 提到质量检查和监控，但没有责任人、阈值、告警或修复闭环。
- 只给出“加资源”或“提高并行度”，没有数据分布和指标证据。

## 5. 风险信号

- 认为 Kafka 不会重复投递，或通过丢弃失败消息保证“只处理一次”。
- 建议关闭 Checkpoint、忽略迟到数据或删除异常分区来恢复任务。
- 无法区分事件时间、处理时间、吞吐和端到端延迟。
- 只罗列 Spark、Flink、Kafka、湖仓框架，不能说明个人职责和数据链路。
- 使用来源不明、口径不一致的数据直接供经营分析或 Agent 使用。

## 6. 建议追问

- 下游数据库不支持事务时，如何实现可接受的幂等？
- Checkpoint 恢复后外部 Sink 如何避免重复写入？
- 热点 Key 会随时间变化时，固定加盐方案有什么风险？
- 实时指标与次日离线结果不一致时，如何定位并校准？
- Schema 变更如何影响 CDC、历史表和下游 Agent？

## 7. 实战任务

设计一条从订单数据库 CDC 到实时经营看板和离线湖仓的链路。方案至少说明：

- Kafka Topic、分区键、消费组和 Offset 策略。
- Flink 事件时间、Watermark、Checkpoint、去重和迟到数据处理。
- 湖仓表分区、主键、增量写入和小文件治理。
- 数据仓库分层、统一指标口径和离线校准。
- 数据质量、血缘、告警、补数与回滚。
- 如何向 Agent 提供经过权限校验且可追踪的数据。

评价应关注数据正确性、恢复能力和工程权衡，不以是否选用某个特定湖仓框架作为淘汰条件。

## 8. 人工复核要求

- 核查候选人所述数据规模、延迟、质量和性能指标的环境与口径。
- 区分本人负责的任务、团队平台能力和已有基础设施。
- 对无法核实的生产经历标记 `UNKNOWN`，安排技术追问，不由模型推断。
- 最终评价由数据平台面试官与 HR 确认，Agent 不得自动推进或淘汰候选人。
