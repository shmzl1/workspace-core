# AI 平台与模型服务工程师结构化面试标准

## 岗位信息

- 公司：云智数科有限公司
- 岗位编号：JOB-AI-PLATFORM-001
- 部门：AI 基础设施部

## 1. 核心能力维度

- Provider 与模型网关：协议适配、路由、权限、配额和版本管理。
- 推理服务工程：部署、容量、资源、健康检查、扩缩容和回滚。
- 稳定性：超时、限流、重试、熔断、隔离、切换和降级。
- 可观测与成本：日志、指标、Trace、告警、Token 和费用治理。
- 模型与 RAG 质量：自动评测、人工评测、召回、忠实度和引用准确性。

## 2. 结构化面试问题

1. 模型接口分别返回 400、401、429 和 5xx 时，你如何分类处理，哪些情况不应重试？
2. 如何设计支持豆包、DeepSeek、通义及自部署模型的统一模型网关？
3. 如何实现模型切换和故障降级，并避免切换后输出契约或向量维度不兼容？
4. 如何监控模型延迟、成功率、Token 和费用，哪些指标用于告警与容量规划？
5. 如何部署和扩缩容推理服务，如何处理模型加载时间、显存和流量突增？
6. 如何管理模型版本、灰度发布、兼容性验证和回滚？
7. 如何评估 RAG 的召回率、忠实度和引用准确性，并区分检索问题与生成问题？
8. 如何避免一个模型故障导致整个 Agent 工作流失败？

## 3. 优秀回答特征

- 将 400 识别为请求或能力不兼容、401/403 识别为认证权限问题，避免盲目重试；429 使用限流和退避，5xx 使用有限重试、熔断与切换。
- 网关包含 Provider Adapter、统一 Schema、路由、认证、配额、审计、超时和错误码映射，上层不依赖供应商细节。
- 模型切换前验证上下文、结构化输出、工具能力和安全策略；Embedding 切换验证维度并规划索引重建。
- 使用成功率、错误类型、P50/P95/P99 延迟、队列、吞吐、Token、单请求成本、降级率和资源利用率建立基线。
- 能说明模型加载、并发批处理、显存、冷启动、健康探针、优雅下线和容量冗余。
- RAG 评测分离检索召回、重排、上下文覆盖、回答忠实度和引用正确性，并保留人工样本复核。

## 4. 一般回答特征

- 能说明重试、监控、灰度和回滚的基本概念，但没有阈值、状态或失败边界。
- 知道 Prometheus、Grafana、Kubernetes 或 vLLM，但不能解释指标、容量和部署拓扑。
- 能调用多个模型 API，但上层代码仍直接依赖各供应商请求结构。
- 把模型评测等同于少量主观问答，没有数据集版本、指标和回归机制。

## 5. 风险信号

- 对所有 4xx、429 和 5xx 无限重试，或在日志中输出 API Key、Authorization Header 和完整敏感输入。
- 模型故障后直接返回成功状态或伪造模型结果，不展示降级与缺失信息。
- 更换 Embedding 模型时忽略向量维度和已有索引兼容性。
- 灰度发布没有流量隔离、监控门禁或回滚条件。
- 将该岗位理解为只编写测试用例，无法说明部署、监控和故障恢复。

## 6. 建议追问

- 如果 400 错误来自“模型不支持当前 API”，网关如何选择正确 Provider Client？
- 429 与客户端超时同时增加时，如何判断是配额、队列还是推理容量问题？
- 熔断后何时半开，如何防止大量请求同时探测恢复？
- 模型版本回滚时，缓存、会话和审计记录如何处理？
- 如何证明一次成本优化没有降低回答质量或 RAG 引用准确性？

## 7. 实战任务

设计一套供企业多 Agent 平台使用的模型服务方案，至少包括：

- 两家云模型和一个自部署推理服务的统一网关与 Provider Adapter。
- 超时、限流、有限重试、熔断、流量隔离和规则降级。
- 模型版本、灰度、切换、回滚及结构化输出兼容性检查。
- 成功率、分位延迟、Token、费用、队列和 GPU/CPU 指标及告警。
- Embedding 与向量索引版本管理。
- 日志脱敏、权限隔离、审计和故障复盘流程。

评价应关注平台稳定性、证据和权衡。模型评测是任务的一部分，但不能替代部署、运行和故障治理能力。

## 8. 人工复核要求

- 面试官需核查候选人是否真正负责过部署、网关、监控或故障处理，而非仅调用模型接口。
- 指标必须记录环境、流量、版本和基线；无法核实的内容标记 `UNKNOWN`。
- 安全方案必须由人工确认，不得因候选人展示敏感日志而视为加分证据。
- Agent 仅整理回答与证据，是否进入下一轮及最终录用由 AI 基础设施面试官和 HR 决定。
