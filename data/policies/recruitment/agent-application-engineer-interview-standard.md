# Agent 应用开发工程师结构化面试标准

## 岗位信息

- 公司：云智数科有限公司
- 岗位编号：JOB-AGENT-001
- 部门：智能应用研发部

## 1. 核心能力维度

- Agent 工作流设计：状态、节点、工具、人工确认、失败恢复和幂等。
- RAG 工程：文档摄取、检索、来源引用、过滤、重排序和明确回退。
- 模型边界：结构化输出校验、超时、限流、降级及确定性结果保护。
- 安全与权限：提示词注入防护、工具最小权限、敏感信息保护和审计。
- 工程交付：FastAPI、PostgreSQL、Docker、可观测和故障复盘。

## 2. 结构化面试问题

1. 一个 Agent Run 在策略节点等待 90 秒后进入回退，你会如何分层排查？
2. RAG 状态显示 `READY`，但业务结果实际使用本地规则回退，可能经过了哪些分支，如何定位？
3. 如何保证模型只能增强叙述，不能修改确定性评分、排名或人工决策？
4. 如何实现多 Agent 状态持久化，使后端重启后能够恢复 Run、节点和事件历史？
5. 如何防止简历中的提示词注入影响系统提示、工具调用或招聘结论？
6. Tool Calling 接入薪资或候选人数据时，权限、输入校验和审计应放在哪里？
7. 模型持续返回不符合 Schema 的 JSON 时，工作流应如何处理？

## 3. 优秀回答特征

- 按客户端、网关、网络、模型、重试配置、节点事件和持久化状态分层定位，而不是直接增加超时时间。
- 能区分健康状态、真实检索调用、元数据过滤、阈值、领域映射和业务回退，并要求回退标识可见。
- 先生成确定性结果，只允许模型更新白名单叙述字段；使用 Schema 校验、不可变字段对比和失败回退。
- 能设计 Run、Node、Event、Tool Call 的稳定标识、顺序、事务和重启恢复策略。
- 将简历视为不可信数据，隔离系统指令，限制工具集合和权限，不执行候选人文本中的命令。
- 能结合本人项目说明故障、指标、权衡和复盘，而非只复述框架能力。

## 4. 一般回答特征

- 能说明超时、重试、数据库和日志的基本作用，但缺少排查顺序或恢复设计。
- 知道 RAG 需要向量检索和引用来源，但不能解释过滤、映射或回退状态。
- 提到 Pydantic、权限或提示词规则，但没有说明如何阻止越权和结果覆盖。
- 项目能够运行演示流程，但缺少异常、审计、部署和指标证据。

## 5. 风险信号

- 建议让模型直接决定录用、淘汰或修改确定性评分。
- 把所有失败都通过无限重试、延长超时或吞掉异常处理。
- 声称 RAG 已命中但无法提供来源、检索条件或实际调用证据。
- 允许 Agent 直接访问数据库、核心算法或高权限工具。
- 把简历内容直接拼入系统提示并允许其改变工具或权限规则。

## 6. 建议追问

- 超时发生后，已经执行的 Tool 是否会重复写入，如何保证幂等？
- SSE 断开重连时，如何补发历史事件并避免顺序错乱？
- 模型输出通过 Schema 但语义与确定性结果冲突时怎么办？
- 更换 Embedding 模型导致维度变化时，如何处理已有索引？
- 请区分你在团队项目中独立完成的模块和他人负责部分。

## 7. 实战任务

给出一个包含策略、知识检索、候选人匹配和报告节点的简化工作流。要求候选人画出状态与调用链，并补充：

- 策略节点超时后的有限重试和规则回退。
- Tool 权限及输入输出 Schema。
- PostgreSQL 状态持久化和事件重放。
- RAG 来源展示与无命中处理。
- 阻止模型覆盖确定性评分的校验点。
- 一条恶意简历指令的防护方案。

评价应依据方案完整性、失败边界、可解释性和候选人对权衡的说明，不要求使用指定 Agent 框架。

## 8. 人工复核要求

- 每个维度必须关联候选人的回答、图示、代码或项目证据，不得由模型补造经历。
- 面试官需确认团队项目中的本人职责，并核查故障与指标口径。
- 信息不足标记 `UNKNOWN` 并记录追问，不直接判定不具备。
- Agent 只能整理证据和提示风险，评分、是否进入下一轮及最终录用由 HR 和业务面试官确认。
