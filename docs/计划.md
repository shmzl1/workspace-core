# TalentFlow 智聘中枢：三 Sprint 开发计划

## Sprint 1：确定性业务核心闭环

### 时间

```text
Day 4 ～ Day 6
```

### Sprint 目标

完成不依赖 LLM 的核心业务闭环，使招聘评分、面试排期、员工考勤、薪资权限和审计均可通过 Web 端实际操作。

Sprint 1 结束时应能演示：

```text
HR 选择岗位
→ 候选人按评分排序
→ 调整权重后排名变化
→ 生成基础面试排期

员工签到
→ 员工签退
→ 查看今日考勤状态

员工查询本人薪资摘要
→ 系统允许

普通员工查询他人薪资
→ 系统拒绝
→ 写入审计日志
```

# Sprint 2：多 Agent 招聘决策主链与实时可视化

## 时间

```text
Day 7 ～ Day 9
```

## Sprint 目标

在 Sprint 1 已完成的确定性招聘评分、面试排期、考勤、薪资和权限闭环基础上，建立招聘决策多 Agent 主链。

Sprint 2 采用以下顶层架构：

```text
企业招聘目标
        ↓
招聘策略 Agent
        ↓
┌──────────────┬──────────────┬──────────────┐
│ 简历解析 Agent │ 岗位匹配 Agent │ 面试评估 Agent │
└──────────────┴──────────────┴──────────────┘
        ↓
决策审查 Agent
        ↓
HR 最终报告
```

Sprint 2 重点完成：

1. 招聘策略 Agent 总控流程。
2. 简历解析 Agent。
3. 岗位匹配 Agent。
4. 面试评估 Agent 的接口和基础结构。
5. 决策审查 Agent。
6. HR 招聘决策报告基础版。
7. 前端实时展示 Agent 执行过程。
8. 候选人比较、证据链、知识来源和可信度展示。
9. 修改企业招聘目标后重新评估并改变候选人排名。

Sprint 2 不要求一次性完成全部小程序和最终部署内容。

---

## 多 Agent 总体职责

### 企业招聘目标

企业招聘目标作为整个多 Agent 工作流的输入，至少包括：

- 岗位。
- 部门。
- 目标招聘人数。
- 招聘截止日期。
- 必备技能。
- 加分技能。
- 最低经验要求。
- 评分阈值。
- 可信度阈值。
- 招聘紧急程度。
- 可选薪资预算。

---

### 招聘策略 Agent

招聘策略 Agent 是整个招聘工作流的总控 Agent。

负责：

1. 读取企业招聘目标。
2. 读取当前岗位和候选人范围。
3. 生成本次招聘分析计划。
4. 决定需要调用哪些专业 Agent。
5. 向简历解析、岗位匹配和面试评估 Agent 分派任务。
6. 汇总企业当前人才供给情况。
7. 将专业 Agent 结果交给决策审查 Agent。
8. 请求生成 HR 最终报告。

招聘策略 Agent 不直接给候选人最终分数，也不自动录用或淘汰候选人。

---

### 简历解析 Agent

负责从候选人简历和现有数据库字段中提取可验证事实。

输出至少包括：

- 原始技能。
- 标准化技能。
- 工作经验。
- 项目经历。
- 项目职责。
- 使用技术。
- 可量化成果。
- 教育背景。
- 证书。
- 到岗时间。
- 信息缺失项。
- 简历证据。

简历解析 Agent 不负责最终岗位评分。

---

### 岗位匹配 Agent

岗位匹配 Agent 内部完成：

```text
企业知识检索
→ 岗位评分标准生成
→ 必备条件检查
→ 技能匹配
→ 经验匹配
→ 项目匹配
→ 调用确定性评分 Service
→ 生成岗位匹配结果
```

输出至少包括：

- 综合评分。
- 岗位匹配度。
- 技能得分。
- 项目得分。
- 经验得分。
- 教育背景得分。
- 风险维度得分。
- 必备条件通过情况。
- 已匹配技能。
- 缺失技能。
- 简历证据。
- 企业知识来源。
- 建议面试问题。

最终数值评分仍由确定性评分算法产生，Agent 不直接随意生成总分。

---

### 面试评估 Agent

Sprint 2 先完成统一输入输出接口和基础流程。

输入可包括：

- 面试状态。
- 面试官结构化评分。
- 技术能力评价。
- 项目能力评价。
- 沟通表达评价。
- 问题解决能力评价。
- 面试风险备注。
- 面试官结论。

若候选人尚未面试，必须返回：

```text
状态：待面试
结论：暂无面试评估结果
```

不得伪造面试评价。

Sprint 3 再完成面试评估 Agent 与真实面试结果、智能排期和最终决策的完整联动。

---

### 决策审查 Agent

负责检查多个 Agent 结果之间是否一致。

重点审查：

- 简历高分项是否有证据。
- 岗位必备条件是否真实满足。
- 企业知识是否使用当前有效版本。
- 岗位匹配结论是否与确定性评分一致。
- 简历信息是否存在缺失或矛盾。
- 面试结果是否与简历描述冲突。
- 多个 Agent 是否存在明显意见分歧。
- 是否需要 HR 人工复核。

决策审查 Agent可以：

- 降低结果可信度。
- 增加风险标签。
- 标记需要人工复核。
- 建议补充面试。
- 建议补充简历材料。

决策审查 Agent不得静默修改确定性总分。

---

### HR 最终报告

HR 最终报告至少包括：

- 企业招聘目标。
- 候选人综合排名。
- 候选人评分分解。
- 简历证据。
- 企业知识来源。
- 必备条件通过情况。
- 面试评估状态。
- Agent 意见分歧。
- 决策审查结果。
- 可信度。
- 推荐动作。
- 建议面试问题。
- 企业当前人才缺口。

最终录用、淘汰和薪资确认仍由 HR 人工完成。

## 阶段目标

- 目标招聘人数。

- 合格候选人数。
- 高可信候选人数。
- 必备条件通过人数。
- 已进入面试人数。
- 尚缺人数。
- 招聘需求满足率。
- 关键技能缺口。
- 岗位供需状态。

```text
评分达到岗位阈值
并且必备条件通过
并且可信度达到阈值
```

供需状态：

- 人才短缺。
- 供给偏紧。
- 基本满足。
- 人才充足。
- 未配置招聘目标。

---

## 招聘策略 Agent 量化建议

招聘策略 Agent 根据实时数据生成：

- 当前供给是否足够。
- 还缺多少合格候选人。
- 最稀缺的岗位技能。
- 哪些候选人应优先推进。
- 哪些候选人需要补充面试。
- 是否需要扩大候选人来源。
- 是否可以调整加分要求。
- 哪些必备条件不能降低。

招聘策略 Agent只给建议，不自动修改企业招聘目标。

---

| 岗位 | 目标人数 | 合格供给 | 高可信供给 | 已面试 | 尚缺人数 | 满足率 | 技能缺口 |
| ---- | -------: | -------: | ---------: | -----: | -------: | -----: | -------- |

所有指标必须来自当前数据库和企业目标数据。

---

## 阶段验收

```text
1. 招聘目标可以结构化展示。
2. 合格供给由评分、必备条件和可信度共同决定。
3. 尚缺人数和满足率来自实时数据。
4. 重新评估后指标合理变化。
5. 面试完成后高可信供给和推荐结论合理变化。
```

## 时间

```text
Day 11
```

## 员工服务 Agent

员工服务 Agent 支持：

- 查询本人今日考勤。
- 查询本人本月考勤。
- 查询本人年假余额。
- 查询本人薪资摘要。
- 查询考勤制度。
- 查询请假制度。
- 查询薪资查看规则。

实时流程：

```text
识别员工身份
→ 判断查询意图
→ 调用业务Tool
→ 必要时检索企业制度
→ 返回业务结果和来源
```

只能查询当前登录员工本人数据。

---

## 小程序定位

微信小程序只用于员工日常操作便利，不复制完整Web管理端。

小程序只接入：

- 员工首页。
- 签到。
- 签退。
- 今日考勤。
- 本月考勤摘要。
- 年假余额。
- 本人薪资摘要。
- 简单制度查询。

小程序不接入：

- HR招聘管理。
- 企业招聘目标。
- 候选人评分。
- 多 Agent 招聘决策。
- 面试排期。
- 面试评价。
- 招聘报告。
- 薪资预审。
- 薪资确认。
- 审计后台。
- Gradio调试台。

---

## 小程序要求

1. 与Web共用同一套FastAPI后端。
2. 不在小程序内自行计算考勤、年假或薪资。
3. API地址从本地配置读取。
4. 不写死真实局域网IP。
5. 只能访问本人数据。
6. 网络失败有明确提示。
7. 权限不足有明确提示。
8. 无数据有明确空状态。
9. Web与小程序数据保持一致。
