## 3. 候选人池

这是当前截图中正在显示的页面。

### 前端主页面

```text
frontend/src/views/CandidateDetailView.vue
```

### 前端请求模块

```text
frontend/src/shared/api/modules/recruitment.ts
frontend/src/shared/api/types.ts
```

调用接口包括：

```http
GET  /recruitment/jobs
GET  /recruitment/candidates
GET  /recruitment/applications
GET  /recruitment/applications/{id}
POST /recruitment/applications/{id}/score
POST /recruitment/applications/{id}/advance
```

### 后端入口

```text
backend/app/api/v1/endpoints/recruitment.py
```

### 后端业务模块

```text
backend/app/modules/recruitment/
├── service.py
├── repository.py
├── models.py
└── schemas.py
```

接口层把请求交给 `RecruitmentService`，Service 再通过 `RecruitmentRepository` 读取：

- 岗位数据。
- 候选人数据。
- 申请数据。
- 招聘阶段记录。

### 简历评分核心算法

```text
backend/app/human_only/resume_scoring.py
```

当前评分维度包括：

- 技能匹配。
- 工作经验。
- 项目经历。
- 教育背景。
- 风险或综合素质。

算法根据权重计算：

- 综合评分。
- 岗位匹配率。
- 风险标签。
- 评分依据。
- 推荐动作。

它目前是确定性规则算法，不是大模型自由生成分数。

评分完成后，后端会把以下内容保存到候选人申请记录中：

- 总分。
- 各维度得分。
- 岗位匹配度。
- 权重快照。
- 评分时间。

### 页面按钮的实际作用

| 页面功能         | 实际执行位置                     |
| ---------------- | -------------------------------- |
| 选择岗位         | 前端按`jobId` 过滤             |
| 智能筛选         | 前端按匹配度和评分组合排序       |
| 按 AI 评分排序   | 前端按已有评分降序排序           |
| 只看高匹配候选人 | 前端筛选匹配度不低于 90%         |
| 点击查看评估     | 请求申请详情接口                 |
| 调整评分权重     | 先保存在前端，重新评估时传给后端 |
| 查看评估依据     | 调用后端评分接口                 |
| 推进阶段         | 调用后端阶段推进接口             |
| 安排面试         | 跳转面试日历，并携带申请 ID      |

智能筛选、排序和高匹配过滤本身并没有运行后端 Agent，只是前端数组计算。

真正重新计算评分的调用流程是：

```text
scoreCandidate()
→ POST /recruitment/applications/{id}/score
→ RecruitmentService.score_application()
→ backend/app/human_only/resume_scoring.py
→ 保存评分到数据库
```

### 为什么截图右侧评分维度全是 0

候选人尚未评分，或者数据库中的 `score_breakdown` 没有对应字段时，前端会把维度值默认设为 `0`。

另外，当前“需复核风险”统计的是所有风险级别不为 `low` 的候选人。

尚未评分的候选人也会暂时被视为 `medium`，所以可能出现：

```text
当前候选人：4
需复核风险：4
```

### 前端

```text
frontend/src/views/InterviewsView.vue
frontend/src/shared/api/modules/interview.ts
```

前端会同时加载：

- 候选人申请。
- 候选人信息。
- 岗位信息。
- 面试官信息。
- 会议室信息。
- 已保存的面试记录。

### 前端接口

```http
GET  /interviews
GET  /interviews/interviewers
GET  /interviews/meeting-rooms
POST /interviews/schedule/preview
POST /interviews/schedule/confirm
```

### 后端

```text
backend/app/api/v1/endpoints/interview.py

backend/app/modules/interview/
├── service.py
├── repository.py
├── models.py
└── schemas.py

backend/app/human_only/interview_scheduler.py
```

### 排期预览流程

```text
InterviewsView.vue
→ generateInterviewSchedule()
→ POST /interviews/schedule/preview
→ InterviewService.preview_schedule()
→ backend/app/human_only/interview_scheduler.py
```

后端会把以下数据交给排期算法：

- 候选人可用时间。
- 面试官可用时间。
- 会议室可用时间。
- 已有面试记录。
- 面试资源冲突情况。

### 确认面试安排

点击“确认面试安排”后，后端会：

1. 再次检查时间是否合法。
2. 检查候选人是否可用。
3. 检查面试官是否可用。
4. 检查会议室是否可用。
5. 检查是否与已有面试冲突。
6. 保存面试记录。
7. 自动推进候选人的招聘阶段。

月视图、周视图和上下月切换属于前端日历计算。

## 6. 薪资明细

### 前端

```text
frontend/src/views/PayrollView.vue

frontend/src/shared/api/modules/
├── payroll.ts
├── payrollReview.ts
├── attendance.ts
└── employee.ts
```

页面会读取：

- 当前账号或目标员工的薪资。
- 月度考勤统计。
- 薪资预审记录。
- 员工列表。

### 后端薪资查询

```text
backend/app/api/v1/endpoints/payroll.py

backend/app/modules/payroll/
├── repository.py
├── models.py
├── schemas/
└── services/
    ├── employee_salary_service.py
    └── access_service.py

backend/app/human_only/salary_access_control.py
```

### 查询流程

```text
PayrollView.vue
→ GET /payroll/me
  或 GET /payroll/employee/{id}
→ EmployeeSalaryService
→ PayrollAccessService
→ backend/app/human_only/salary_access_control.py
→ PayrollRepository / SalaryRecord
→ 返回脱敏后的薪资数据
```

权限算法会根据以下信息进行判断：

- 当前账号角色。
- 当前账号权限。
- 是否查询本人。
- 是否查询直接下属。
- 是否查询同部门员工。
- 是否拥有薪资管理员权限。

判断结果包括：

- 能否查看。
- 哪些字段可见。
- 哪些字段需要脱敏。

每次允许或拒绝的访问都会写入审计日志。

### 当前注意事项

页面中的迟到、早退、缺勤扣款金额目前是在前端按固定规则计算，例如：

```text
迟到一次扣 50 元
早退一次扣 50 元
缺勤按日薪扣除
无薪假按日薪扣除
```

“导出 PDF”目前尚未实现，只显示提示。

## 10. 系统设置

```text
frontend/src/views/SettingsView.vue
```

这是一个纯前端展示页面：

- 智能解析开关没有状态绑定。
- 简历评分开关没有状态绑定。
