# AI反思日志

# 2026年7月6日

## 功能点一：项目范围与架构确定

### 问了什么
向AI询问企业HR管理Agent项目的模块划分，以及如何在有限周期内兼顾功能完整度和技术深度。

### AI给了什么
AI建议将项目划分为招聘管理、员工服务、考勤管理、薪资预审、权限审计、Agent工作流、企业制度RAG、Web管理端、微信小程序和Gradio调试台十个模块，并采用 `API → Service → Repository → PostgreSQL` 分层架构，Agent通过 `Agent → Tool → Service` 调用业务层。

### 我们实际改了什么
1. 确定了十大模块范围，Gradio定位为内部调试工具。
2. 统一使用FastAPI后端支撑Web、小程序和Gradio。
3. 确立分层架构，Agent不直接访问数据库，通过Tool调用Service。
4. 根据项目周期删除了短期无法完成的功能。

### 使用反思
AI的分层建议和模块划分作为起点是合理的，但最终取舍必须由团队根据实际工期的判断。AI不知道团队的技术储备和演示优先级，它列出的模块清单不能直接当作最终计划。

## 功能点二：开发环境与分支规范

### 问了什么
向AI询问四人团队如何统一开发环境、数据库端口、Git分支和前后端启动方式。

### AI给了什么
AI建议：日常开发使用 `dev` 分支、稳定版合并到 `main`；Docker启动PostgreSQL；Alembic管理数据库迁移；前后端分别维护环境变量示例；在README中记录环境配置。

### 我们实际改了什么
1. 日常开发统一在 `dev` 分支，`main` 作为稳定分支。
2. PostgreSQL通过Docker Compose启动，数据库结构用Alembic维护。
3. 修正了数据库端口和后端环境变量不一致的问题。
4. 在README中明确了数据库、后端和前端的启动顺序。

### 使用反思
环境规范这类问题AI能给出一套标准答案，但团队实际落地时需要逐条验证——比如AI推荐的端口可能与本地已有服务冲突，需要团队根据实际情况调整。

## 功能点一：候选人池评分实时化

### 问了什么
向AI询问候选人池评分为什么基本固定为81%，以及如何让评分随数据库数据实时变化。

### AI给了什么
AI建议检查完整调用链：前端是否用固定数据 → API是否调用Service → Service是否从数据库读取 → 是否调用评分算法 → 字段是否一致，并建议将评分拆分为多维度。

### 我们实际改了什么
1. 删除前端固定评分逻辑，改为从数据库读取。
2. 统一评分接口和前端字段（`overall_score`、`match_score`、风险标签、评分依据等）。
3. 评分算法不可用时展示明确错误信息，不再用固定分数掩盖失败。

### 使用反思
AI的调用链排查思路有效，但这种问题本质上是前后端联调不充分导致的——AI只能指出排查方向，真正修复需要开发者在浏览器Network面板中逐层验证。

## 功能点二：面试日历与智能排期联调

### 问了什么
向AI询问面试日历一直加载、智能排期无结果的问题根源。

### AI给了什么
AI建议检查前端传参（候选人编号vs申请编号）、数据库读取可用时间、三方共同时间段计算、已有面试冲突检查和预览/确认字段一致性。

### 我们实际改了什么
1. 统一前端的排期入口使用 `application_id`。
2. 后端从数据库读取候选人、面试官、会议室和已有面试。
3. 排期算法计算三方可用时间交集并排除冲突时段。
4. 排期确认前再次检查冲突，修复面试日历加载状态无法结束的问题。

### 使用反思
排期联调中字段不一致是最常见的问题来源，AI能列出一系列可能的错误点，但必须由开发者在实际调试中定位具体是哪个环节出了问题。

## 功能点三：招聘报告实时化

### 问了什么
向AI询问招聘报告如何根据岗位、候选人和申请数据自动更新，而不是展示固定统计。

### AI给了什么
AI建议后端统一汇总岗位数、候选人分布、阶段分布、部门情况、月度趋势和平均评分，前端只展示不独立维护统计数据。

### 我们实际改了什么

### 使用反思
实时化改造的核心是后端汇总接口设计，AI在这个问题上的建议比较直接——只要数据链路打通，报告自然实时化。

## 功能点二：面试排期算法与薪资访问控制理解

### 问了什么
向AI询问排期算法和薪资访问控制的核心逻辑和边界场景。

### AI给了什么
AI解释了排期算法的时间解析、交集计算、冲突判断和方案评分流程，以及薪资访问控制中本人/全量/部门/脱敏四种权限的字段覆盖差异。

## 功能点一：多Agent招聘架构重构

### 问了什么
向AI询问如何将招聘评分、简历解析、岗位匹配、面试评估、决策审查和报告组织成可解释的多Agent工作流。

### AI给了什么
AI建议建立招聘策略、简历解析、岗位匹配、面试评估、决策审查和HR报告六个节点，每个节点记录输入、输出、状态、耗时和来源。

### 我们实际改了什么
1. 重构多Agent流程：策略节点生成执行计划，简历解析只提取事实不虚构信息。
2. 岗位匹配调用确定性评分Service，无真实面试评价时标记为 `SKIPPED`。
3. 决策审查检查评分、证据、风险和可信度，HR报告只汇总不直接决定录用。
4. 最终决定保留给HR人工完成。

### 使用反思
AI给出的节点设计思路合理，但"什么节点应该标记SKIPPED而非FAILED"这类决策实际上隐含了业务判断——AI无法区分哪些缺失是合理的（如尚无面试记录）、哪些是系统异常。

## 功能点二：Agent调用边界与SSE实时展示

### 问了什么
向AI询问Agent如何调用业务数据同时不绕过Service和权限边界，以及前端如何实时展示Agent执行过程和刷新恢复。

### AI给了什么
AI建议使用Tool封装业务能力并注入给Runner，Agent架构为 `Agent → Tool → Service → Repository`（涉及核心算法时经过 `human_only`）；前端方面建议生成 `run_id` 和 `trace_id`，通过SSE推送事件，保存Run Snapshot，刷新后通过GET接口恢复，事件ID去重。

### 我们实际改了什么
1. 确立Agent调用链，Tool只负责调用Service不复制业务算法。
2. 建立Agent Run、节点、事件和Tool调用的数据结构，状态保存到PostgreSQL。
3. SSE推送真实Agent事件，历史事件从数据库重新读取。
4. 刷新后通过Snapshot恢复，不同用户间Run隔离，事件不暴露完整异常。

### 使用反思
SSE和Snapshot的设计是本次最关键的架构决策之一。AI能给出完整的技术方案，但并发锁超时清理、异常断流恢复等细节仍需要在实测中验证和修复。

## 功能点一：LLM与RAG接入及回退机制

### 问了什么
向AI询问如何为招聘策略和HR报告接入大模型与企业知识检索，同时保证模型不可用时系统仍能运行。

### AI给了什么
AI建议增加模型网关、Embedding客户端、ChromaDB向量存储和知识检索网关，并设计超时和本地规则回退机制，以及健康检查接口。

### 我们实际改了什么
1. LLM和RAG作为可选集成，未配置模型时不阻塞业务启动。
2. 策略和报告先生成确定性结果，再尝试模型增强。
3. 模型失败时使用 `RULE_BASED_FALLBACK`，RAG不可用时使用 `LOCAL_HYBRID_FALLBACK`。
4. 前端明确显示真实集成状态，健康检查隐藏密钥和本地路径。

### 使用反思
AI对回退机制的设计思路很有价值——"永远有兜底"是Agent系统的基本原则。但AI建议中的ChromaDB和模型网关的具体配置细节仍需开发者根据实际部署环境调整。

整理AI反思日志的过程本身就是一种反思——我们发现AI有时会"过度承诺"（把尚未验证的建议描述得像已完成的方案），读者（包括答辩评委）应能通过日志中的"AI给了什么"和"我们实际改了什么"之间的差异来理解这些建议哪些最终落地、哪些只是参考。

# 2026年7月15日

## 功能点一：前端认证、权限路由与公共错误处理架构审查

### 问了什么
向AI询问 `authStore.ts`、`LoginView.vue`、`guards.ts`、`apiClient.ts` 和 `AttendanceView.vue` 的代码组织方式及协作机制。

### AI给了什么
AI将每个文件拆解为具体的组织单元：`authStore.ts` 为Vue3 Composable + reactive单例模式，`guards.ts` 为5步判断链，`apiClient.ts` 为axios实例+请求/响应双层拦截器，`AttendanceView.vue` 为状态机模式。AI还绘制了协作关系图，指出 `apiClient` 作为唯一HTTP门面的设计理念。

### 我们实际改了什么
1. 确认前端认证采用无Pinia依赖的Composable单例模式。
2. 确认路由守卫权限校验为OR逻辑，与路由表 `meta.anyPermissions` 对应。
3. 确认 `apiClient` 响应拦截器剥离 `{success, data}` 后调用方拿到纯净业务数据。
4. 确认401处理链路的双向兜底（拦截器 + guards.ts）和错误区分链路（403 vs 其他）。
5. 将审查结论与团队共享，确保架构理解一致。

### 使用反思
AI对代码组织方式的拆解很细致，直接定位到每个函数和类型的职责。但AI的分析基于代码快照——如果存在未发现的bug或未覆盖的边界，AI不会主动指出。AI的协作关系图也不应直接放入正式架构文档，而应作为内部理解辅助。

## 功能点二：Sprint 2 测试覆盖与边界场景验证

### 问了什么
向AI询问如何验证后端核心模块（招聘评分、面试排期、薪资访问控制、Agent运行、考勤）的测试覆盖是否充分。

### AI给了什么
AI建议按模块盘点：评分（空候选人/空岗位/权重异常）、排期（无可用时间/三方无交集/会议室全冲突）、薪资（四种权限字段差异）、Agent（并发锁/SSE断流/跨用户隔离）、考勤（重复签到/未签退/跨天签退）、权限路由（未登录/无权限URL/token过期）。

### 我们实际改了什么
1. 整理了各模块测试覆盖清单，标记已覆盖和缺失场景。
2. 为招聘评分3个边界场景补充单元测试，为排期2个冲突场景增加测试数据。
3. 为Agent并发锁和跨用户隔离补充集成测试。
4. 将边界场景与"真实缺陷、潜在风险、一般优化"三级分类对齐。

### 使用反思
AI能列出常见边界场景清单，但无法判断哪些场景已有代码处理只是缺测试、哪些在当前架构下根本不成立（如Pydantic已在后端校验）。不能将AI列出的所有场景不加区分地写入测试计划——应先逐条确认是否可能发生。

## 功能点三：深色模式适配与响应式布局修复

### 问了什么
向AI询问深色模式切换时部分页面颜色不一致（如琥珀色标签在深色背景下仍显示浅色背景），以及移动端卡片溢出和按钮重叠的排查修复方法。

### AI给了什么
AI建议：检查硬编码颜色在 `[data-theme="dark"]` 下的覆盖、审计全局CSS变量的双主题值、使用DevTools移动端模拟器逐页检查、将深色覆盖规则集中管理。

### 我们实际改了什么
1. 修复了 `AttendanceView.vue` 中琥珀色标签在深色模式下的颜色异常。
2. 修复了 `LoginView.vue` 登录页面的深色模式适配（背景、标题、输入框、演示账号区域）。
3. 检查了候选人池、面试日历和薪资页面的深色模式渲染。
4. 对招聘仪表板布局重构，确保响应式断点下的卡片排列合理。
5. 保持 `[data-theme="dark"]` 覆盖规则在各组件scoped style底部集中。

### 使用反思
AI在CSS适配方面的建议偏通用——它无法知道组件使用了哪些具体颜色类、哪些已被全局变量覆盖。AI给出的CSS代码不能直接复制使用，实际修复需开发者在浏览器中逐页切换主题、对比差异、手动调整。AI的价值在于提供排查思路而非直接生成修复代码。

## 功能点四：多Agent评估界面重构验收与候选人池完善

### 问了什么
向AI询问多Agent评估界面重构后的功能完整性和候选人池中简历预览、上传和评分同步功能的验收。

### AI给了什么
AI建议逐项验收：组件拆分是否完成、`run_id` 是否写入URL、SSE事件是否正确更新、候选人池的简历操作是否正常、四种Agent节点状态是否有对应UI展示。

### 我们实际改了什么
1. 完成组件拆分验收，确认各面板独立渲染。
2. 验证 `run_id` 写入URL后刷新可恢复状态，SSE事件去重逻辑生效。

### 使用反思
AI的验收清单有助于系统性检查完整性。但需警惕：AI列出的验收项部分基于之前描述的功能设计而非实际代码——如果开发中做了调整但未告知AI，验收清单就会脱节。最终验收必须以浏览器实际操作和状态观察为准。
