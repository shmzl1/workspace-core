## 四、遇到的问题与困难

### 1. 技术问题

- 项目前端页面较多，不同模块最初存在布局、字段命名、状态表示和错误提示不统一的问题。
- 非 Agent 模块涉及前端、API、Service、Repository 和 PostgreSQL，多层数据结构不一致时定位问题较复杂。
- 薪资权限既要满足不同角色的访问规则，又要保证字段脱敏、拒绝反馈和审计记录同步生效。
- Sprint 2 的企业知识需要同时考虑岗位、部门、版本和生效时间，过期规则不能错误地作为当前制度使用。
- 多 Agent Trace 和前端实时展示需要提供足够的过程信息，同时不能泄露完整简历、薪资数据和内部敏感内容。

### 2. 已采取的解决方案

- 统一前端组件、页面布局、接口封装、状态枚举和错误反馈规范。
- 按照“前端请求—API—Service—Repository—数据库”的顺序逐层排查数据链路。
- 将薪资访问统一收口到 `access_service` 和薪资权限禁飞区，避免其他模块绕过权限判断直接读取数据。
- 对允许、拒绝和敏感字段访问统一记录审计日志。
- 为企业知识增加版本、生效时间和适用岗位等元数据，并设计当前有效版本优先的检索规则。
- 对 Agent 事件和 Trace 使用结构化摘要，只展示当前目标、Tool 调用、来源和结果摘要，不返回完整敏感内容。

### 3. 需要帮助或待解决

无。

### 4. 非技术问题

- 实际开发过程中承担了较多跨模块任务，前端整体设计、业务联调和权限模块之间需要合理安排时间。
- 部分实际贡献与原始计划分工存在交叉，需要通过 Git 提交记录和周报说明明确各自完成内容。

### 5. 已采取的措施

- 按功能模块拆分任务，优先保证 Sprint 1 核心链路可演示，再推进 Sprint 2 企业知识与 Agent 相关内容。
- 保留关键功能的 Git 提交记录，记录页面、接口、数据库和禁飞区代码的修改范围。
- 与其他成员明确边界，不重复声明招聘评分、智能排期算法和多 Agent 招聘主链等由其他成员负责的核心工作。

- 系统地学习了 AI Agent 的工程化开发流程，理清了 Naive RAG、Advanced RAG、Graph RAG 等检索增强生成范式以及 LangChain 等开发框架之间的架构关联与适用场景，掌握了在多 Agent 协同工作流中组织角色、绑定 Tool 并进行状态流转的设计思路。

### 2. 对项目和开发流程的思考

项目不能只关注单个页面或单个接口是否可用，还需要保证前端、后端、数据库、权限和审计形成完整闭环。尤其是薪资等敏感业务，不能仅在前端隐藏按钮，而必须在后端统一进行权限校验，并对允许和拒绝行为保留审计记录。

在多 Agent 系统中，企业知识必须具有明确来源、版本和生效时间，Agent 只能引用当前有效制度。前端展示也不能使用固定日志或模拟过程，而应与真实后端事件保持一致。

下周将进一步加强企业知识检索准确性、权限边界和异常场景测试，同时继续整理个人代码提交与功能说明，使每项贡献都能够通过页面、接口、数据库记录和代码调用链进行验证。
