## 四、遇到的问题与困难

### 技术问题

#### 1. 深色模式下多组件样式继承与优先级冲突

全站深色模式涉及 34 个文件的样式调整，部分组件使用了不同层级的样式定义（全局 CSS 变量、Scoped SCSS、内联样式和第三方组件库默认样式），存在样式优先级冲突，导致部分区域在主题切换后出现色彩不一致。

**已采取的解决方案：**
- 建立统一的 CSS 自定义属性变量体系（`variables.scss` + `index.css`），将主题色值集中管理。
- 逐组件排查样式来源，统一使用 CSS 变量引用而非硬编码色值。
- 对第三方组件库的深色模式覆盖采用更高优先级的 `:root.dark` 选择器。
- 编写 `useTheme.ts` Composable 统一管理主题切换逻辑，确保所有组件通过一致的接口获取当前主题状态。

#### 2. 决策审查服务圈复杂度过高，难以维护和测试

`decision_review_service.py` 中核心函数承担了候选人评分汇总、面试分析、风险评估、建议生成等多重职责，圈复杂度远超可维护阈值，导致单元测试难以覆盖所有分支路径。

**已采取的解决方案：**
- 将原函数拆分为评分汇总、面试表现分析、风险等级评估和综合建议生成四个独立子模块。
- 每个子模块具备明确的输入/输出接口，便于独立测试和维护。
- 保持对外调用接口不变，确保重构不影响上游 Agent 和 API 层的调用逻辑。
- 通过 Locust 性能测试验证重构后接口性能无明显回退。

#### 3. 考勤审查页面全栈联调中的数据一致性问题

考勤审查功能涉及后端 API、业务服务层、前端展示和状态管理多个环节。前后端在考勤异常类型的枚举定义、审查状态流转规则和日期格式上存在不一致，导致联调初期数据展示异常。

**已采取的解决方案：**
- 统一前后端考勤异常类型枚举（迟到/早退/缺勤/加班异常/正常）。
- 明确审查状态流转规则：待审查 → 已确认 / 已驳回，且驳回后支持重新审查。
- 统一日期格式为 ISO 8601，前后端一致使用 UTC 时区。
- 在前端 API 层增加数据格式转换和校验逻辑，确保后端返回数据异常时前端能优雅降级。

#### 4. 多 Agent 页面的深色模式与动画效果相互影响

多 Agent 评估页面同时应用了入场动画（`reveal-animations.css`）和深色模式样式，部分动画关键帧使用了固定色值，在深色模式下出现闪烁或颜色突变。

**已采取的解决方案：**
- 将动画关键帧中的固定色值替换为 CSS 变量引用。
- 在主题切换时增加短暂过渡（`transition: background-color 0.3s ease, color 0.3s ease`），避免颜色突变。
- 确保动画裁剪容器（`overflow: hidden`）正确继承父元素的圆角设置，与深色模式的高亮层圆角保持一致。

### 对项目和开发流程的思考

Sprint 2 的收尾阶段让我更加深刻地认识到：**技术实现与质量保障同等重要**。在紧张的项目周期中，不仅需要完成功能开发，还需要配套完成代码质量检查、性能测试、兼容性验证和文档沉淀，才能交付一个真正"完成"的功能模块。

深色模式的实现看似是"换一套颜色"，但在实际工程中，它考验的是整个前端样式体系的架构设计——如果早期没有建立统一的变量管理机制，34 个文件的逐一修改将极其痛苦。这让我更加理解"好的架构设计是为未来的变更服务的"。

圈复杂度的治理也不仅仅是"让代码看起来更整洁"，而是直接关系到代码的可测试性、可维护性和 Bug 风险。一个圈复杂度过高的函数，不仅难以编写覆盖全部分支的单元测试，还容易在修改时引入隐蔽的新 Bug。
