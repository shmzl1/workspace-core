## 四、遇到的问题与困难

### 1. 技术问题

#### （1）多 Agent 页面内容较多

多 Agent 页面需要同时展示招聘目标、多个节点、事件记录、候选人结果和最终报告，内容较多时容易显得混乱。

**解决办法：** 将页面分成任务概览、工作流、事件记录、节点详情和结果展示几个部分，并统一不同节点的状态和颜色。

#### （2）手机访问时不能直接使用 localhost

电脑上的 `localhost` 指向电脑本机，但手机上的 `localhost` 指向手机自身，因此手机打开页面后可能无法请求电脑上的后端。

**解决办法：** 让前端和后端监听局域网地址，手机通过笔记本局域网 IP 访问，同时检查接口代理、端口和跨域配置。

#### （3）Agent 事件和页面状态有时不一致

如果前端节点名称和后端事件状态没有对应好，可能出现节点一直显示运行中，或者任务完成后页面没有及时更新。

**解决办法：** 统一节点名称和状态，根据后端返回的真实事件更新页面，并在刷新后重新读取任务信息。

#### （4）不同页面的字段和空数据处理不统一

候选人列表、详情、面试日历和招聘报告会使用相同的数据，但部分页面最初对评分、状态、时间和空值的处理不同。

**解决办法：** 以实际接口返回为准统一字段，区分加载中、无数据、请求失败、无权限和正常显示几种状态。

**需要帮助或待解决：** 无。

### 2. 非技术问题

本周需要同时处理前端界面、手机访问、接口联调、项目检查和答辩准备，涉及内容较多，时间安排比较紧张。

**已采取的措施：** 先处理影响主要演示流程的问题，再完善样式和文档；同时和组员确认接口和任务分工，避免重复修改。

- 对登录权限、薪资字段脱敏和审计日志的作用有了更清楚的认识。

### 2. 对项目和开发流程的思考

多 Agent 页面不只是把几个 Agent 节点画出来，更重要的是让用户看清任务运行到哪一步、每个节点做了什么以及最终结果是什么。前端应该展示真实的后端状态，不能为了页面效果使用固定日志或模拟数据。

手机访问项目看起来只是更换访问地址，但实际还涉及前端服务、后端服务、接口代理、端口和局域网环境。只有这些配置保持一致，手机才能正常打开页面并加载数据。

在开发业务系统时，无数据、数据为零、请求失败和没有权限是不同的情况，需要给用户不同提示。页面是否真实、清楚，比单纯追求视觉效果更重要。

### 3. 自我改进方面

以后开发复杂页面时，我会先确认接口字段和页面状态，再开始编写组件，减少后期联调时的重复修改。同时会更早检查手机端、网络异常、权限不足和空数据等情况，并及时整理相关文档。
