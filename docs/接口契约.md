# 接口契约

本文记录规划中的 API 契约与 Sprint 1 已生成的基础路由。不表示接口已完成联调或测试通过。

当前数据库 ORM、首次迁移文件、基础 Service/Repository 和部分只读 API 已建立。数据库迁移、种子导入和接口联调仍需人工在本地环境执行。

## API 前缀

统一前缀：`/api/v1`

`GET /health` 保留 `status=ok`，并返回安全的 `overall_mode`、`run_store.mode`、`integrations.llm` 与 `integrations.rag` 状态。RAG 状态包含 Collection、文档数和 Chunk 数；接口不返回 API Key、Token、数据库连接串、本地绝对路径或模型原始响应。LLM/RAG 禁用时普通业务仍可用，最近调用或初始化失败时整体状态为 `DEGRADED`。

## 统一响应结构

```json
{
  "success": true,
  "data": {},
  "error": null,
  "trace_id": "request-trace-id"
}
```

## 错误返回

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "ERROR_CODE",
    "message": "错误说明"
  },
  "trace_id": "request-trace-id"
}
```

## 身份上下文

- 除登录和健康检查外，业务接口只接受 `Authorization: Bearer <JWT>`。
- JWT 用于确认用户身份；每次请求仍从 `users` 表读取账号状态、`role` 和 `permissions`。
- `permissions` 是最终接口授权依据；角色用于展示、审计和默认初始化，不替代权限判断。
- `401` 表示 Token 缺失、无效、过期或账号停用；`403` 表示身份有效但权限不足。
- Web 与小程序共享员工相关 API。
- 小程序不能调用 HR 招聘、薪资预审、审计后台接口。

## 登录与权限

`POST /api/v1/auth/login` 请求：

```json
{ "username": "zhangwei", "password": "password" }
```

成功响应的 `data` 包含 `access_token`、`token_type`、`expires_in` 与 `user`。
`user` 包含 `id`、`username`、`role`、`permissions`、关联 `employee_id`、姓名、部门和职位。
`GET /api/v1/auth/me` 返回同一用户结构，用于页面刷新时恢复会话。

接口权限使用稳定代码，例如 `candidate.score`、`interview.manage`、`audit.read`、
`payroll.all.read`；薪资读取还会校验员工关系并按字段范围返回。前端菜单隐藏仅改善交互，不能替代接口授权。

## Sprint 1 基础路由

- `GET /health`：健康检查，返回统一响应和 `trace_id`。
- `POST /api/v1/auth/login`：校验用户名和密码哈希，返回 JWT 与当前用户权限。
- `GET /api/v1/auth/me`：返回 JWT 对应账号的最新角色、权限和员工档案摘要。
- `GET /api/v1/employees`：读取员工列表。
- `GET /api/v1/employees/me`：读取当前登录员工档案。
- `GET /api/v1/employees/me/leave-balance`：读取当前登录员工年假余额。
- `GET /api/v1/recruitment/jobs`：读取岗位列表。
- `GET /api/v1/recruitment/candidates`：读取候选人列表。
- `POST /api/v1/recruitment/candidates/import`：接收一个或多个 PDF 简历；岗位信息只能从简历文本提取并与开放岗位精确匹配，不接收 `job_id`。
- `GET /api/v1/recruitment/jobs/{job_id}`：读取单个岗位完整信息。
- `GET /api/v1/recruitment/jobs/{job_id}/applications`：读取岗位候选人投递与排序字段。
- `GET /api/v1/recruitment/applications/{application_id}`：读取申请、候选人、岗位、保存的评分和阶段流水。
- `POST /api/v1/recruitment/applications/{application_id}/advance`：推进候选人阶段并写入阶段流水。
- `GET /api/v1/interviews`：读取已排面试列表。
- `GET /api/v1/interviews/interviewers`：读取面试官列表。
- `GET /api/v1/interviews/meeting-rooms`：读取会议室列表。
- `POST /api/v1/interviews/schedule/preview`：从数据库资源和可用时间生成排期预览。
- `POST /api/v1/interviews/schedule/confirm`：再次校验资源冲突后保存面试安排。
- `GET /api/v1/attendance/today`：读取当前登录员工今日考勤。
- `GET /api/v1/attendance/records`：读取当前登录员工近期考勤。
- `GET /api/v1/payroll/me`：读取本人薪资，必须经过薪资权限 Service。
- `GET /api/v1/payroll/employee/{employee_id}`：读取指定员工薪资，必须经过权限和数据范围校验。
- `GET /api/v1/audit/logs`：读取审计日志列表。

## 规划接口分组

- `/auth`：登录、当前身份、角色上下文。
- `/employees/me`：员工本人信息。
- `/attendance`：签到、签退、今日考勤、本月摘要。
- `/leave`：年假余额和请假摘要。
- `/policies`：制度查询和来源展示。
- `/payroll/me`：员工本人薪资摘要查询。
- `/payroll-review`：HR 薪资预审、待 HR 确认、确认与拒绝。
- `/recruitment`：岗位、候选人、评分、比较和流程。
- `/interviews`：面试排期、会议室、面试官和冲突说明。
- `/audit`：敏感访问和操作审计。
- `/agent`：Agent 任务提交、Trace 和结果查询。

## 边界

- 员工本人薪资查询必须经过 `salary_access_control` 权限校验。
- 薪资读取同时校验账号权限、员工关系和字段范围；允许与拒绝结果均写入审计日志。
- HR 薪资预审和 HR 薪资确认是不同动作。
- 签到与签退只记录时间事实，不做位置验证。
- Agent 查询必须返回 Trace 摘要，RAG 问答必须返回来源。
- API 层通过 Service 和 Repository 访问数据库，不得在路由中直接编写业务查询。

# Sprint 1 联调字段补充

- `GET /api/v1/recruitment/applications` 的列表项包含
  `candidate_name` 与 `job_title`，前端以 `application.id` 作为排期选择值。
- `GET /api/v1/interviews/interviewers` 除面试官资源字段外返回
  `employee_name`，用于展示真实面试官姓名。
- `POST /api/v1/interviews/schedule/preview` 请求字段为 `application_id` 和可选
  `duration_minutes`；候选人、面试官、会议室、可用时间和已有面试均由 Service 从数据库读取。
- `POST /api/v1/interviews/schedule/confirm` 请求包含 `application_id`、`interviewer_id`、
  `meeting_room_id`、`start_at`、`end_at` 与 `conflict_explanation`。Service 会在保存前再次检查
  同一候选人、面试官或会议室的时间冲突，并在成功后通过招聘 Service 推进申请阶段。
- `GET /health` 用于前端页面加载前的后端连通性检查，不属于 `/api/v1` 前缀。
- `GET /api/v1/recruitment/report?time_range=30d|90d|all` 从招聘岗位、候选人和申请记录
  汇总核心指标、招聘漏斗、部门统计、候选人来源和月度趋势。空库返回零值和空数组。
- 招聘评分写回 `candidate_applications.score_breakdown` 时包含 `match_score` 和
  `overall_score`、各维度原始分、实际归一化权重及加权贡献；`weights_snapshot` 保存本次实际使用的权重。
  报告与候选人池从同一评分记录读取数据。

## Sprint 2.3 确定性招聘 Agent Run

以下接口均要求 JWT 和 `agent.hr.use` 权限：

| 方法     | 路径                                               | 权限                              | 请求                      | 响应                                          | 当前状态 | 错误码                   | 人工验收 |
| -------- | -------------------------------------------------- | --------------------------------- | ------------------------- | --------------------------------------------- | -------- | ------------------------ | -------- |
| `POST` | `/api/v1/agent/recruitment/runs`                 | JWT +`agent.hr.use`             | `RecruitmentRunRequest` | `ApiResponse<RecruitmentRunSnapshot>`       | 代码存在，待本地人工验收 | `CANDIDATE_NOT_IN_JOB` | 需要     |
| `GET`  | `/api/v1/agent/recruitment/runs/{run_id}`        | JWT +`agent.hr.use`，仅创建用户 | 路径参数`run_id`        | `ApiResponse<RecruitmentRunSnapshot>`       | 代码存在，待本地人工验收 | `AGENT_RUN_NOT_FOUND`  | 需要     |
| `POST` | `/api/v1/agent/recruitment/runs/{run_id}/approve-job-match-review` | JWT +`agent.hr.use`，仅创建用户 | 路径参数`run_id`，无请求体 | `ApiResponse<RecruitmentRunSnapshot>` | 代码存在，待本地人工验收 | `RECRUITMENT_REVIEW_NOT_PENDING`（409） | 需要 |
| `POST` | `/api/v1/agent/recruitment/runs/{run_id}/approve-decision-review` | JWT +`agent.hr.use`，仅创建用户 | 路径参数`run_id`，无请求体 | `ApiResponse<RecruitmentRunSnapshot>` | 代码存在，待本地人工验收 | `RECRUITMENT_REVIEW_NOT_PENDING`（409） | 需要 |
| `GET`  | `/api/v1/agent/recruitment/runs/{run_id}/events` | JWT +`agent.hr.use`，仅创建用户 | 路径参数`run_id`        | `text/event-stream`，裸 `AgentEvent` JSON | 代码存在，待本地人工验收 | `AGENT_RUN_NOT_FOUND`  | 需要     |

POST 请求为 `RecruitmentRunRequest`。后端校验候选人不能为空、候选人真实存在且已投递当前岗位，并使用数据库岗位名称和部门规范化目标。错误码包括：

- `AGENT_CANDIDATES_REQUIRED`
- `JOB_NOT_FOUND`
- `CANDIDATE_NOT_IN_JOB`
- `AGENT_RUN_NOT_FOUND`

SSE 格式：

```text
id: <event_id>
event: agent_event
data: <AgentEvent JSON>
```

`RecruitmentRunSnapshot` 保留 `created_at`、`updated_at`，并包含规范化后的 `goal`、`job`、`candidate_ids`、`current_candidate_id`、候选人进度、节点状态、执行计划、`candidate_profiles`、`job_rubric`、`knowledge_summary`、`job_matches`、`interview_evaluations`、`decision_reviews`、`report` 和 `sources`。前后端字段均保持 `snake_case`。

当前执行计划的 `current_phase` 为 `SPRINT_2_3_DETERMINISTIC_INTERMEDIATE`，`next_phase` 为 `LLM_RAG_INTEGRATION`；`executed_nodes` 为 `recruitment_strategy`、`resume_parser`、`job_match`、`decision_review`、`hr_report`，`skipped_nodes` 只包含 `interview_evaluation`。无真实结构化面试评价时，跳过原因为 `STRUCTURED_INTERVIEW_FEEDBACK_NOT_AVAILABLE`，决策审查仍继续运行。

岗位匹配结果通过 `INTERMEDIATE_RESULT.summary.job_match_summary` 发送，决策审查结果通过 `REVIEW_COMPLETED.summary.decision_review` 发送，最终报告通过 `REPORT_GENERATED.summary.report` 发送。节点可终止为 `COMPLETED`、`NEEDS_REVIEW` 或 `SKIPPED`；`NEEDS_REVIEW` 是业务复核结果，不等同于工作流失败。

失败事件的公共错误仅包含安全的错误码、说明、`failed_node`、`failed_step` 和异常类型。

三个既有 Run API 的路径、JWT、`agent.hr.use`、所有者隔离、`ApiResponse` 和 SSE 裸 `AgentEvent` 格式保持兼容。Run、节点、事件和 Tool 调用由 PostgreSQL Store 持久化；LLM 只增强策略与报告叙述，RAG 正常时返回 `CHROMA_HYBRID`，禁用或失败时返回 `LOCAL_HYBRID_FALLBACK`。代码存在，待本地人工验收。

当岗位匹配或决策审查节点为 `NEEDS_REVIEW` 时，Run 保持 `RUNNING` 且 `hr_report` 保持 `WAITING`。仅 Run 所有者可分别调用 `approve-job-match-review` 和 `approve-decision-review`；每个操作只记录对应节点的 HR 审查通过并将其标记为 `COMPLETED`，不会修改评分或审查发现。两个待审节点都确认后才继续生成 HR 报告；重复审批或非待审状态返回 409。

本次没有数据库结构或环境变量变更，因此没有 Alembic 同步命令或 `.env` 同步项。
