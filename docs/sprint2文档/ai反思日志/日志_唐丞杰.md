# AI反思日志

> 本文档以"问了什么 → AI给了什么 → 我们实际改了什么 → 使用反思"四段式记录 2026年7月6日至7月15日的AI辅助开发过程。
> 项目名称为 TalentFlow 智聘中枢，团队四人（唐丞杰、吴越、刘子恒、黄钧），采用一套 FastAPI 后端支撑 Vue Web 管理端、微信小程序员工端和 Gradio 内部调试台。
> 项目设定了三个 AI 禁飞区（`backend/app/human_only/` 下的 `resume_scoring.py`、`interview_scheduler.py`、`salary_access_control.py`），由人工编写和维护核心算法，AI 不得修改。
> Sprint 1/2 期间完成了 CH1（性能优化）、CH4（测试强化）、CH5（架构重构）三项 AI 抗性挑战。

# 2026年7月6日

## 功能点一：项目范围与架构确定

### 问了什么
向AI询问企业HR管理Agent项目的模块划分，以及如何在有限周期内兼顾功能完整度和技术深度。

### AI给了什么
AI建议将项目划分为招聘管理、员工服务、考勤管理、薪资预审、权限审计、Agent工作流、企业制度RAG、Web管理端、微信小程序和Gradio调试台十个模块，并采用 `API → Service → Repository → PostgreSQL` 分层架构，Agent通过 `Agent → Tool → Service` 调用业务层。

### 我们实际改了什么
1. 确定了十大模块范围，Gradio定位为内部调试工具，不作为正式业务入口。
2. 统一使用FastAPI后端支撑Web、小程序和Gradio，一套后端覆盖所有前端入口。
3. 确立分层架构：普通业务 `API → Service → Repository → PostgreSQL`，Agent调用 `Agent → Tool → Service → human_only`。
4. 根据项目周期删除了短期无法完成的功能（不做第二套后端、微服务、Redis、Celery、RabbitMQ、Kubernetes；不做真实微信OAuth、公网部署；薪资模块不做银行发薪/税务/社保/公积金）。
5. 明确了系统角色：普通员工、HR专员、部门主管、薪酬管理员，不同角色对应不同功能边界。

### 使用反思
AI的分层建议和模块划分作为起点是合理的，但最终取舍必须由团队根据实际工期判断。AI不知道团队的技术储备和演示优先级，它列出的模块清单不能直接当作最终计划。特别地，需求说明中"不做项"清单（如不做第二套后端、不做GPS考勤等）是团队基于时间和演示场景做出的主动取舍，这些决策需要人工判断而非AI建议。

## 功能点二：开发环境与分支规范

### 问了什么
向AI询问四人团队如何统一开发环境、数据库端口、Git分支和前后端启动方式。

### AI给了什么
AI建议：日常开发使用 `dev` 分支、稳定版合并到 `main`；Docker启动PostgreSQL；Alembic管理数据库迁移；前后端分别维护环境变量示例；在README中记录环境配置。

### 我们实际改了什么
1. 日常开发统一在 `dev` 分支，`main` 作为稳定分支。
2. PostgreSQL通过Docker Compose启动，数据库结构用Alembic维护（首次迁移 `0001_initial_schema`，只创建结构不写入种子数据）。
3. 修正了数据库端口和后端环境变量不一致的问题，统一使用 `.env.example` 模板（`backend/.env.example`、`frontend/.env.example`）。
4. 在 `docs/environment-setup.md` 中记录了数据库、后端和前端的启动顺序。

### 使用反思
环境规范这类问题AI能给出一套标准答案，但团队实际落地时需要逐条验证——比如AI推荐的端口可能与本地已有服务冲突，需要团队根据实际情况调整。数据库基线只负责结构，业务API、Service、Repository需要按模块逐步补齐。

## 功能点二：数据库结构设计

### 问了什么
向AI询问项目需要哪些数据库表来覆盖员工、招聘、面试、考勤、薪资、权限、审计和Agent运行记录。

### AI给了什么
AI给出了用户、员工、岗位、候选人、申请、面试、考勤、薪资、制度、审计和通知等实体建议，并建议外键表达业务关系、权限以稳定字符串代码保存、评分和排期保存输入快照。

### 我们实际改了什么
1. 建立了覆盖主要业务的22张数据库表：用户账号与员工档案分开；岗位、候选人和候选人申请三层关系；面试官、会议室和面试记录表；考勤相关表（`work_calendars`、`attendance_records`、`leave_balances`）；薪资相关表（`salary_records`、`payroll_periods`、`payroll_rules`、`payroll_adjustments`、`payroll_review_records`、`payroll_line_items`）；审计表（`audit_logs`）。
2. 数据库迁移只负责表结构（Alembic `0001_initial_schema`），演示数据通过独立脚本（`scripts/build-demo-data.py`、`scripts/seed-data.ps1`）导入。
3. 评分结果保存评分明细（`score_breakdown`）和权重快照（`weights_snapshot`），避免权重变化影响历史评分。

### 使用反思
AI给出的实体模型基本覆盖了业务需求，但一些关联表的细节（如面试官和会议室的关联方式）仍需团队根据实际业务逻辑补充和调整。ORM模型集中在 `backend/app/modules/*/models.py`，`backend/app/core/database.py` 只提供 `Base`、通用时间戳、引擎和会话工厂。

## 功能点一：候选人池评分实时化

### 问了什么
向AI询问候选人池评分为什么基本固定为81%，以及如何让评分随数据库数据实时变化。

### AI给了什么
AI建议检查完整调用链：前端是否用固定数据 → API是否调用Service → Service是否从数据库读取 → 是否调用评分算法 → 字段是否一致，并建议将评分拆分为多维度。

### 我们实际改了什么
1. 删除前端固定评分逻辑（`CandidateDetailView.vue` 中的硬编码数据），改为从PostgreSQL读取候选人和申请记录。
2. 统一评分接口和前端字段：`overall_score`（综合评分）、`match_score`（岗位匹配度）、`skill_match`（技能匹配说明）、`risk_tags`（风险标签）、`risk_prompt`（风险提示）、`scoring_basis`（评分依据）、`score_breakdown`（分项评分）、`recommended_action`（推荐动作）。
3. 评分算法不可用时展示 `HUMAN_ONLY_ALGORITHM_NOT_READY` 或等价说明，不再用固定分数掩盖失败。
4. 评分完成后写回 `candidate_applications` 的 `score_breakdown`，包含各维度原始分、实际归一化权重及加权贡献。

### 使用反思
AI的调用链排查思路有效，但这种问题本质上是前后端联调不充分导致的。评分调用链路为：`CandidateDetailView.vue → POST /recruitment/applications/{id}/score → RecruitmentService.score_application() → human_only/resume_scoring.py → 保存评分到数据库`。AI只能指出排查方向，真正修复需要开发者在浏览器Network面板中逐层验证。

## 功能点二：面试日历与智能排期联调

### 问了什么
向AI询问面试日历一直加载、智能排期无结果的问题根源。

### AI给了什么
AI建议检查前端传参（候选人编号vs申请编号）、数据库读取可用时间、三方共同时间段计算、已有面试冲突检查和预览/确认字段一致性。

### 我们实际改了什么
1. 统一前端的排期入口使用 `application_id`，不再混用 `candidate_id`。
2. 后端从数据库读取候选人可用时间、面试官可用时间、会议室可用时间和已有面试记录。
3. 排期接口 `POST /api/v1/interviews/schedule/preview` 调用 `InterviewService.preview_schedule()` 再委托 `human_only/interview_scheduler.py` 的 `schedule_interview()` 函数，计算三方可用时间交集并排除冲突时段。
4. 排期确认（`POST /api/v1/interviews/schedule/confirm`）再次校验资源冲突后保存面试安排，并自动推进候选人招聘阶段。
5. 修复面试日历加载状态无法结束的问题，月/周视图基于浏览器本地日期和后端ISO时间生成。

### 使用反思
排期联调中字段不一致是最常见的问题来源——`application_id` vs `candidate_id` 的混用导致了大量"无数据"问题。AI能列出一系列可能的错误点，但必须由开发者在实际调试中定位具体是哪个环节出了问题。确认流程中的"预览-确认分离"设计是关键：预览只返回排期建议，确认时才真正写入数据库，且确认前再次检查冲突。

## 功能点三：招聘报告实时化

### 问了什么
向AI询问招聘报告如何根据岗位、候选人和申请数据自动更新，而不是展示固定统计。

### AI给了什么
AI建议后端统一汇总岗位数、候选人分布、阶段分布、部门情况、月度趋势和平均评分，前端只展示不独立维护统计数据。

### 我们实际改了什么
1. 增加招聘报告数据汇总接口 `GET /api/v1/recruitment/report?time_range=30d|90d|all`，从数据库实时计算。
2. 报告与候选人池读取同一份评分记录（`score_breakdown` 中的 `overall_score` 和 `match_score`），支持不同时间范围查询。
3. 空数据库返回零值和空数组，删除前端固定统计数据（`frontend/src/shared/data/recruitmentDashboard.ts` 中的硬编码数据改为从API获取）。

### 使用反思
实时化改造的核心是后端汇总接口设计，AI在这个问题上的建议比较直接——只要数据链路打通（岗位→候选人→申请→评分记录全部来自PostgreSQL），报告自然实时化。不过当前报告中的"AI洞察"部分仅是前端文本模板，尚未接入大模型。

## 功能点一：候选人评估失败问题排查

### 问了什么
向AI询问后端已有评分代码但前端仍显示"无法获取智能评估"的排查路径。

### AI给了什么
AI建议依次检查：网络请求、API路径与前缀（`/api/v1`）、JWT与权限、响应格式（`{success, data, error, trace_id}`）、字段是否在 `data` 内、编号是否混用（`candidate_id` vs `application_id`）、异常是否被统一转换。

### 我们实际改了什么
1. 对齐前端API地址和后端路由，修正候选人编号与申请编号的使用。
2. 统一后端响应中的评分字段，修正前端读取 `overall_score`、`match_score` 和 `score_breakdown` 的位置。
3. 分开处理401（Token缺失/无效/过期）、403（身份有效但权限不足）、业务错误和网络错误，不把所有异常都显示为"无法连接网络"。
4. 当禁飞区模块不可加载或未实现时，接口返回 `algorithm_not_ready`，页面展示明确错误，不隐藏失败。

### 使用反思
AI给出的排查清单本质上是"按调用链路从外到内逐级打日志"的思路。最终定位往往是一个小的字段名不一致——但AI无法替开发者跑浏览器DevTools，排查过程仍需人工完成。统一响应格式 `{success, data, error, trace_id}` 是所有接口遵守的契约，前端 `apiClient.ts` 的响应拦截器在剥离 `{success, data}` 包装后，业务调用方拿到的 `response.data` 已是纯净数据，无需重复判断 `success` 字段。

## 功能点二：前端业务状态统一与Sprint 1 Bug修复

### 问了什么
向AI询问候选人池、面试日历、招聘报告、薪资和考勤页面如何统一处理加载、空数据、权限不足和后端错误状态。

### AI给了什么
AI建议为页面统一设计Loading、Empty、Error、PermissionDenied、正常业务五种状态，并提供重试和Toast提示。

1. 为主要页面增加加载状态（`LoadingState` 组件）、空数据提示（`EmptyState`）、权限拒绝组件（`PermissionDenied`）和错误状态（`ErrorState` + 重试按钮）。
2. 修复了11个Sprint 1 Bug：
   - **P0阻塞级**：BUG-S1-001（AppShell导入缺失mock文件导致编译失败）、BUG-S1-002（各视图使用硬编码静态数据未接入API层）、BUG-S1-003（共享反馈组件未被任何视图使用）——三大核心问题全部修复，五大核心页面（Attendance/Payroll/CandidateDetail/Interviews/Audit）均通过 `shared/api/modules/*` 调用后端API，具备完整状态切换。
   - **P1严重级**：BUG-S1-004（候选人评分页缺少权重调整交互，新增5维滑块沙盘）、BUG-S1-005（考勤签到/签退按钮无实际交互反馈，新增checkIn/checkOut逻辑+loading+Toast）、BUG-S1-006（面试排期页无冲突可视化，新增红色边框+标签的冲突样式）、BUG-S1-007（审计日志无搜索/筛选和展开详情功能）——全部修复并通过回归。
   - **P2一般级**：BUG-S1-009（"查看预审详情"按钮无导航）、BUG-S1-011（移动端权重滑块布局优化，添加响应式媒体查询）——已修复。BUG-S1-008（次要视图仍为静态占位）——不予修复（Sprint 2范围）。BUG-S1-010（权限拒绝态测试）——已验证。
3. 删除后端失败后自动显示固定业务数据的做法，不显示"Sprint 1""开发中"等过程性文字。

### 使用反思
AI给出的五状态模式是前端开发的通用实践，但它无法感知哪些页面当前缺少哪种状态处理。Sprint 1 Bug清单中的11个问题全部由人工在浏览器中逐页操作发现和验证——AI无法自动发现"按钮没有点击事件"或"组件未被引用"这类体验问题。P2级别的BUG-S1-008（次要视图静态占位）被标记为"不予修复"并划到Sprint 2，这个决定也是基于团队对优先级的判断，而非AI建议。

# 2026年7月10日

## 功能点一：招聘评分算法理解与边界检查

### 问了什么
向AI询问评分算法中技能、经验、项目、学历和风险各项如何参与总分计算。

### AI给了什么
AI逐步解释了输入读取、技能标准化、经验年限、项目匹配、学历分数、风险标签、权重归一化和加权总分计算流程，并指出需检查缺失字段、非法权重和异常数据类型。

### 我们实际改了什么
1. 重新检查 `backend/app/human_only/resume_scoring.py` 中 `score_resume(payload: dict) -> dict` 的每个辅助函数。
2. 确认权重总和异常时进行归一化，空输入提供稳定返回（返回 `status: "algorithm_not_ready"` 而非崩溃）。
3. 保留评分依据（`scoring_basis` 列表）和加权贡献，识别必备技能（`required_skills`）和加分技能（`preferred_skills`）不应完全合并的问题——后续Review中确认缺少加分技能不应触发"关键技能缺失"风险。
4. 补充对异常 `profile_json` 类型（字符串/列表而非字典）的处理计划。
5. 确认算法输入仅接收结构化 `dict`（包含 `job`、`candidate`、`weights` 三个键），不依赖 FastAPI、SQLAlchemy 或 LLM。

### 使用反思
AI对算法代码的逐函数解释有助于快速理解逻辑，但不能替代对数学正确性的验证——评分权重和归一化逻辑是否合理，仍需要领域知识判断。`human_only_contracts.md` 中约定的输入输出结构是Service层与禁飞区之间的接口契约，AI虽然能解释已有代码，但无法验证契约的实现是否正确。

## 功能点二：面试排期算法与薪资访问控制理解

### 问了什么
向AI询问排期算法和薪资访问控制的核心逻辑和边界场景。

### AI给了什么
AI解释了排期算法的时间解析、交集计算、冲突判断和方案评分流程，以及薪资访问控制中本人/全量/部门/脱敏四种权限的字段覆盖差异。

### 我们实际改了什么
1. 检查 `backend/app/human_only/interview_scheduler.py` 中 `schedule_interview(payload: dict) -> dict` 的输入输出：输入包含 `application_id`、`candidate.available_slots`、`interviewers`、`meeting_rooms`、`duration_minutes`；输出包含 `recommended_time`、`recommended_interviewer_id`、`recommended_room_id`、`conflict_detection`、`recommendation_reason`。
2. 检查无可用时间和缺少资源时排期的返回，保留最多五个候选排期方案；识别面试时长非法（零值/负数）、带时区/不带时区时间混用的问题。
3. 明确薪资访问判断和薪资预审是两种不同业务：`check_salary_access(payload: dict) -> dict` 当参数为角色字符串时执行字段级访问判断，当参数为字典时返回薪资预审摘要——同一函数两种返回结构，后续Review中计划拆分为 `check_salary_access()` 和 `review_salary_records()` 两个独立函数。

### 使用反思
AI能快速梳理代码逻辑，但发现不了架构层面的设计问题——比如 `salary_access_control.py` 中同一函数同时承担访问判断和预审摘要两种职责（根据参数类型返回不同结构），只有开发者结合业务需求才能察觉这种设计是否合理。该问题已在后续7月13日的交叉Review中被正式提出。

## 功能点一：多Agent招聘架构重构

### 问了什么
向AI询问如何将招聘评分、简历解析、岗位匹配、面试评估、决策审查和报告组织成可解释的多Agent工作流。

### AI给了什么
AI建议建立招聘策略、简历解析、岗位匹配、面试评估、决策审查和HR报告六个节点，每个节点记录输入、输出、状态、耗时和来源，并将"招聘决策Agent"定位为整个工作流而非单体Agent。

### 我们实际改了什么
1. 重构多Agent流程，确立六节点架构：
   - **招聘策略Agent**：读取企业招聘目标与候选人范围，生成 `RecruitmentExecutionPlan`（包含 `executed_nodes` 和 `skipped_nodes`），不直接生成评分。
   - **简历解析Agent**：确定性提取 `CandidateProfile` 与 `ResumeEvidenceItem`，只提取已有事实不虚构信息；简历内指令只作待分析文本。
   - **岗位匹配Agent**：调用确定性评分Service（`DETERMINISTIC_HUMAN_ONLY` 模式），输出 `JobMatchSummary` 与技能/项目缺口。
   - **面试评估Agent**：只使用真实结构化面试数据；无数据时以 `STRUCTURED_INTERVIEW_FEEDBACK_NOT_AVAILABLE` 标记为 `SKIPPED`，不生成虚假评价。
   - **决策审查Agent**：按规则检查无证据高分、必备条件、来源版本、面试数据缺失，`review_mode` 为 `RULE_BASED_INTERMEDIATE`，`deterministic_score_preserved` 始终为 `true`。
   - **HR最终报告**：按确定性规则汇总排名与建议，`requires_human_decision` 始终为 `true`，不拥有录用/淘汰/排期确认/薪资确认权。
2. 当前阶段为 `SPRINT_2_3_INTEGRATED`，执行五个节点（面试评估跳过），下一阶段为 `END_TO_END_VALIDATION`。

### 使用反思
AI给出的六节点设计思路合理，但"什么节点应该标记SKIPPED而非FAILED"这类决策隐含业务判断——比如 `NEEDS_REVIEW` 是合法业务结果，不会单独使工作流失败。AI无法区分哪些缺失是合理的（如尚无面试记录）、哪些是系统异常。Agent只提供事实、证据、审查与建议，HR保留录用、淘汰、排期确认和薪资确认权——这个边界是团队主动设计的，AI只是按描述给出架构。

## 功能点二：Agent调用边界与SSE实时展示

### 问了什么
向AI询问Agent如何调用业务数据同时不绕过Service和权限边界，以及前端如何实时展示Agent执行过程和刷新恢复。

### AI给了什么
AI建议使用Tool封装业务能力并注入给Runner，Agent架构为 `Agent → Tool → Service → human_only`（涉及核心算法时经过 `human_only`）；前端方面建议生成 `run_id` 和 `trace_id`，通过SSE推送事件，保存Run Snapshot到PostgreSQL，刷新后通过GET接口恢复，事件ID去重。

### 我们实际改了什么
1. 确立Agent调用链：Agent不访问Repository或 `human_only`；Tool只调用Service，不创建新数据库Session；岗位匹配/决策审查/报告Tool只委托对应Service。
2. 建立Agent Run、节点、事件和Tool调用的数据结构（`RecruitmentRunSnapshot` 包含 `run_id`、`created_at`、`goal`、`job`、`candidate_ids`、节点状态、`candidate_profiles`、`job_matches`、`decision_reviews`、`report` 等），状态保存到PostgreSQL（`PostgreSQLAgentRunStore`）。
3. SSE推送真实 `AgentEvent`（使用稳定大写枚举和 `snake_case` 字段，`AGENT_THINKING` 只包含可审计结构化阶段摘要，不暴露隐藏思维链/密码/JWT/完整薪资）。
4. 三个Run API：`POST /api/v1/agent/recruitment/runs`（创建Run，含JWT+`agent.hr.use`权限校验，候选人非空且已投递当前岗位）、`GET /api/v1/agent/recruitment/runs/{run_id}`（恢复Snapshot，仅Run创建者可读）、`GET /api/v1/agent/recruitment/runs/{run_id}/events`（SSE历史事件重放）。
5. 不同用户间Run隔离，失败事件只返回安全的 `failed_node`、`failed_step`、异常类型和公共错误说明。

### 使用反思
SSE和Snapshot的设计是本次最关键的架构决策之一。AI能给出完整的技术方案，但几个关键细节仍需在实测中验证：①并发锁（`_run_locks`）的超时清理逻辑（当前 `cleanup_expired()` 固定返回0，无实际清理）；②SSE单个事件解析失败可能中断整个事件流（后续Review中提出需对每个事件块单独 `try/catch`）；③启动时将未终止的 `PENDING/RUNNING` Run标记为 `FAILED`（`AGENT_RUN_INTERRUPTED_BY_RESTART`），当前阶段不自动续跑。

## 功能点一：LLM与RAG接入及回退机制

### 问了什么
向AI询问如何为招聘策略和HR报告接入大模型与企业知识检索，同时保证模型不可用时系统仍能运行。

### AI给了什么
AI建议增加模型网关（OpenAI-compatible Chat Client）、Embedding客户端（支持标准 `/embeddings` 和火山方舟 `/embeddings/multimodal`）、ChromaDB向量存储、知识检索网关，并设计超时（招聘策略25秒、HR报告35秒）和本地规则回退机制，以及健康检查接口。

2. 策略和报告先生成确定性结果，再通过OpenAI-compatible Gateway增强白名单叙述字段；模型不可用时使用 `RULE_BASED_FALLBACK` 标记，继续使用已生成的确定性计划或报告。
3. RAG正常时返回 `CHROMA_HYBRID`，禁用/失败时使用 `LOCAL_HYBRID_FALLBACK`（位于招聘Service中的企业知识本地混合回退），不冒充ChromaDB命中。

### 使用反思

向AI询问 `authStore.ts`、`LoginView.vue`、`guards.ts`、`apiClient.ts` 和 `AttendanceView.vue` 的代码组织方式及协作机制。

### AI给了什么
AI将每个文件拆解为具体的组织单元：`authStore.ts` 为Vue3 Composable + reactive单例模式（`useAuthStore()` + `readCachedUser()`/`setSession()`/`clearSession()` + `login()`/`logout()`/`hasPermission()`/`hasAnyPermission()`），`guards.ts` 为5步判断链（`setupGuards(router)` → `router.beforeEach`），`apiClient.ts` 为 `axios.create()` 实例+请求拦截器（自动附加Bearer Token）+响应拦截器（两层：成功时剥离 `{success, data}` 包装、错误时按401/403/网络断开分别处理并统一包装为 `ApiClientError` 类），`AttendanceView.vue` 为状态机模式（loading → permissionDenied → pageError → 正常）。

### 我们实际改了什么
1. 确认前端认证采用无Pinia依赖的Composable单例模式，`authStore.ts` 中模块级 `reactive(state)` 单例保证跨组件状态共享。
2. 确认路由守卫权限校验为OR逻辑（`hasAnyPermission`），与路由表 `meta.anyPermissions` 字段一一对应；`getDefaultRoute()` 按权限决定登录后跳转。
3. 确认 `apiClient.ts` 响应拦截器在剥离 `{success, data}` 包装后，所有业务调用方拿到的 `response.data` 已是纯净业务数据，无需重复判断 `success` 字段。
4. 确认401处理链路的双向兜底：响应拦截器清除localStorage+强制跳转 `/login`，与 `guards.ts` 的未认证拦截形成双重保障。
5. 确认 `AttendanceView.vue` 的错误处理链路：API请求 → 拦截器抛出 `ApiClientError` → catch块按 `error.status === 403` 区分权限拒绝/其他错误 → 分别设置 `permissionDenied`/`pageError`。
6. 同时确认了7月14日Review中发现的四个问题的修复方向：网络异常不应清除会话（401/Token无效/过期时才清）、无权限跳转应携带提示、考勤应使用专用权限码、页面请求应解耦非关键请求。

### 使用反思
AI对代码组织方式的拆解很细致，直接定位到每个函数和类型的职责。但AI的分析基于代码快照——7月14日Review中发现的 `loadCurrentUser()` 中网络异常导致强制退出问题，AI在最初的分析中并未指出。AI的协作关系图不应直接放入正式架构文档，而应作为内部理解辅助。

## 功能点二：Sprint 2 测试覆盖与边界场景验证

### 问了什么
向AI询问如何验证后端核心模块（招聘评分、面试排期、薪资访问控制、Agent运行、考勤）的测试覆盖是否充分。

### 我们实际改了什么
1. 整理了各模块测试覆盖清单，标记已覆盖场景和缺失场景。现有的 `backend/tests/human_only/` 下三个禁飞区测试文件（`test_resume_scoring.py`、`test_interview_scheduler.py`、`test_salary_access_control.py`）已覆盖高匹配、技能缺失、经验不足、时间冲突、无交集等基础情况。

3. 为排期2个冲突场景增加测试数据：三方无交集返回明确不可排期、会议室全冲突返回冲突说明。

5. 将边界场景与"真实缺陷、潜在风险、一般优化"三级分类对齐：7月13-14日Review中发现的8个问题按此三级分类归档。
6. 确认考勤签到/签退的重复操作和异常时序场景已纳入测试范围。

### 使用反思
AI能列出常见边界场景清单，但无法判断哪些场景已有代码处理只是缺测试、哪些在当前架构下根本不成立（如Pydantic已在后端校验、前端不会再收到非法类型）。不能将AI列出的所有场景不加区分地写入测试计划——应先逐条确认是否可能发生。另外，`docs/测试覆盖.md` 中记录当前 `4 failed, 12 passed` 的边界异常测试结果（参见 `docs/边界处理.md` 中的 guardrails `startswith` 修复），修复后回归 `16 passed`。

### 问了什么

### AI给了什么
AI建议：检查硬编码颜色在 `[data-theme="dark"]` 下的覆盖、审计全局CSS变量的双主题值（`--color-*`）、使用DevTools移动端模拟器逐页检查、将深色覆盖规则集中管理。

### 我们实际改了什么
1. 修复了 `AttendanceView.vue` 中琥珀色标签（`bg-amber-50`、`text-amber-600`、`bg-amber-500/20`）在深色模式下的颜色异常。
2. 修复了 `LoginView.vue` 登录页面的深色模式适配（背景、标题、输入框、演示账号区域）。
3. 检查了候选人池、面试日历和薪资页面的深色模式渲染。
4. 对招聘仪表板布局重构（`RecruitmentDashboardPage.vue`），确保响应式断点下的卡片排列和间距合理；修复移动端（<860px）候选人评分权重滑块布局（5列改2列）。

## 功能点五：Sprint 1 AI抗性挑战收口 —— CH1（性能优化）、CH4（测试强化）、CH5（架构重构）

### 问了什么
向AI询问在Sprint 2收尾阶段如何验证Sprint 1选择的三项AI抗性挑战（CH1性能优化、CH4测试强化、CH5架构重构）是否达到达标标准。

### AI给了什么
AI建议：
- **CH1**：使用Locust对关键API进行压测，验证响应时间<200ms；识别性能瓶颈。
- **CH4**：盘点单元测试覆盖率和异常路径覆盖情况；结合边界场景分类补充测试。
- **CH5**：通过圈复杂度工具对比重构前后代码复杂度；确认降幅≥50%。

### 我们实际改了什么
1. **CH1性能优化**：使用Locust对全部关键API端点进行压测（`/health`、`/api/v1/auth/me`、`/api/v1/recruitment/*`、`/api/v1/interviews`、`/api/v1/attendance/today`、`/api/v1/audit/logs` 等11个端点）。压测结果显示：除 `/api/v1/attendance/today`（当前377次请求均失败，需排查）外，所有GET端点的中位数响应时间均<26ms，最大响应时间<150ms，达标（参见 `docs/性能检查/locust_report_stats.csv`）。
2. **CH4测试强化**：三个禁飞区测试（`test_resume_scoring.py`、`test_interview_scheduler.py`、`test_salary_access_control.py`）覆盖正常路径和异常路径；`backend/tests/concurrency/test_splitter_concurrency.py` 覆盖并发场景；Sprint 1 Bug清单记录了14个Bug的复现、修复和回归；7月13-14日Review发现的8个问题均已记录并安排修复。
3. **CH5架构重构**：重构了 `decision_review_service.py`（参见 `docs/架构重构挑战/decision_review_service00.py` → `decision_review_service01.py`），将原实现的复杂嵌套逻辑拆分为独立的审查规则函数（`_check_evidence_gaps()`、`_check_prerequisites()`、`_check_source_consistency()`、`_check_interview_data()`、`_compute_credibility()`），每个函数职责单一、圈复杂度显著降低。Sprint 2.3整体架构从单体Agent拆分为六节点工作流，Tool层与Service层明确分离。

### 使用反思
三项AI抗性挑战的核心价值在于：①CH1的性能数据为系统能力提供了量化证据——大部分API在<30ms内响应，验证了FastAPI+PostgreSQL方案的可行性；②CH4的测试覆盖强化证明了"AI可以辅助列边界场景，但最终测试编写和场景确认必须由人工完成"；③CH5的架构重构展示了AI辅助重构的局限性——AI能识别可以拆分的代码块，但无法判断拆分后的职责边界是否合理（如职位匹配和决策审查的输入输出契约需要人工定义）。三项挑战的证据材料分别位于 `docs/性能检查/`、`docs/单元测试.png`+`docs/并发测试.png`、`docs/架构重构挑战/`。
