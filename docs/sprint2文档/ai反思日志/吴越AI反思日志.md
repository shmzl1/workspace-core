# 2026年7月6日

## 功能点一：招聘管理端页面框架搭建

### 问了什么

我向AI询问：招聘管理端首页如何组织导航、数据概览、候选人快捷入口和待办区域，既能覆盖招聘业务，又不让页面信息过于拥挤。

### AI给了什么

AI建议按照“导航区、统计区、待办区、最近候选人区”划分页面，并给出Vue组件拆分建议：

1. 侧边导航组件。
2. 顶部用户与通知区域。
3. 招聘数据统计卡片。
4. 待处理面试和候选人列表。
5. 通用空状态与加载状态组件。

### 我实际改了什么

1. 完成招聘管理端基础布局和菜单结构。
2. 将候选人池、面试日历、招聘报告等入口放入统一导航。
3. 统计卡片先使用明确的展示字段，后续再接入真实接口。
4. 为列表区域预留加载、空数据和错误提示位置。
5. 统一页面标题、卡片圆角、间距和按钮样式。

### 使用反思

AI给出的第一版布局组件层级偏多，部分样式也比较通用，直接使用会增加维护成本。我认识到页面开始前应先确定信息优先级和组件边界，AI更适合帮助完成初版结构，不能代替页面取舍。

## 功能点二：前端项目目录与公共组件整理

### 问了什么

我向AI询问：招聘、考勤、薪资等页面逐渐增多后，前端目录如何分类，才能避免公共组件和业务组件混放。

### AI给了什么

AI建议按页面、业务组件、通用组件、接口请求、状态管理和类型定义分目录管理，并建议提取分页、状态标签、确认弹窗和提示消息等公共能力。

### 我实际改了什么

1. 按业务页面和通用组件整理前端文件。
2. 抽取状态标签、空状态、分页和确认弹窗等可复用部分。
3. 将接口请求集中管理，避免页面内散落请求地址。
4. 为候选人、面试和权限状态补充统一显示文案。

### 使用反思

AI容易把所有组件都过度抽象，导致简单页面也要经过很多层传参。我学到公共组件应服务于真实重复场景，只有样式和行为都稳定时才适合抽取。

# 2026年7月7日

## 功能点一：候选人池列表与筛选交互

### 问了什么

我向AI询问：候选人池页面需要支持姓名搜索、岗位筛选、流程状态筛选和分页，前端应怎样管理筛选条件与列表请求。

### AI给了什么

AI建议把筛选条件集中为查询对象，在搜索、切换筛选项和翻页时统一请求列表；同时建议增加防抖、重置页码和筛选条件回显。

### 我实际改了什么

1. 完成候选人姓名、岗位和状态筛选区域。
2. 列表展示候选人基本信息、申请岗位、匹配分数和当前阶段。
3. 接入分页参数，支持切换页码和每页数量。
4. 筛选条件变化时重置为第一页。
5. 增加无筛选结果时的提示和清空筛选入口。

### 使用反思

AI初版遗漏了“筛选变化后页码仍停留在旧页”的情况，也没有统一参数名称。我认识到列表页的关键不是把控件摆出来，而是明确筛选、分页和请求参数的状态关系。

## 功能点二：候选人详情页信息展示

### 问了什么

我向AI询问：候选人详情页如何展示基础资料、技能标签、评分依据、风险提示和面试记录，才能让HR快速查看重点信息。

### AI给了什么

AI建议使用分区信息卡片，将分数、推荐意见和风险标签放在顶部，其余资料按“基础信息、简历摘要、评估结果、面试记录”排列。

### 我实际改了什么

1. 增加候选人详情的基础资料和岗位申请信息。
2. 展示技能标签、评分明细、匹配度和风险提示。
3. 将面试记录和操作入口放在同一页面，减少跳转。
4. 对缺少头像、简历摘要和面试记录的情况增加默认展示。

### 使用反思

AI曾直接读取可能为空的字段，页面出现过`undefined`。我学到详情页必须把接口缺失、空数组和状态未生成视为正常情况，不能假设所有数据都完整。

# 2026年7月8日

## 功能点一：候选人评分展示与接口联调

### 问了什么

我向AI询问：候选人列表中的评分如何从后端真实返回，并在评分暂未生成、评分失败或分数为空时保持页面表达清晰。

### AI给了什么

AI建议前端区分“已评分、评估中、待评估、评估失败”四类状态；已评分时展示总分和维度摘要，失败时展示可重试提示，不要以固定分数代替真实结果。

### 我实际改了什么

1. 对齐列表和详情页中的评分字段名称。
2. 将总分、匹配度、风险标签和评分理由分别展示。
3. 评分未生成时显示待评估状态，不显示虚构数据。
4. 评分接口异常时展示明确错误信息和重试操作。
5. 候选人池和招聘报告读取同一份评分结果。

### 使用反思

AI一度建议用默认分数保证界面完整，但这会掩盖接口或算法问题。我认识到业务系统中“没有数据”和“数据为零”必须区分，真实状态比页面好看更重要。

## 功能点二：招聘数据概览卡片调整

### 问了什么

我向AI询问：招聘首页的岗位数、候选人数、待面试人数和平均匹配度如何避免使用固定统计数据，并兼顾空数据展示。

### AI给了什么

AI建议通过一个概览接口返回聚合数据，前端只负责格式化展示；空数据时返回零值，趋势图或分布图没有数据时使用空状态代替模拟图表。

### 我实际改了什么

1. 将概览卡片改为读取接口返回数据。
2. 为统计数字增加加载骨架和请求失败提示。
3. 空数据时显示零值及说明，不再放置随机增长趋势。
4. 统一数字、百分比和日期的前端格式化方式。

### 使用反思

AI生成图表时默认补充了模拟数据，容易让演示结果看起来像真实业务。我学到展示层不能为了视觉效果伪造业务信息，尤其是招聘和薪资相关页面。

# 2026年7月9日

## 功能点一：前端请求封装与异常提示

### 问了什么

我向AI询问：多个页面调用招聘接口时，如何统一处理请求地址、登录失效、权限不足、业务错误和网络异常。

### AI给了什么

AI建议建立统一请求封装，在响应拦截处处理令牌和基础错误；页面层只处理当前业务提示，并将401、403、422和网络错误使用不同文案展示。

### 我实际改了什么

1. 集中管理招聘和面试相关接口请求。
2. 对登录失效、权限拒绝、参数错误和网络失败分别提示。
3. 页面请求增加加载状态，防止用户重复提交。
4. 列表和详情请求失败时保留重试入口。
5. 修正部分接口前缀和参数名不一致的问题。

### 使用反思

AI初版把多数异常都归为“网络错误”，会误导排查方向。我认识到前端错误提示不仅是用户体验，也影响定位问题的效率，必须保留足够的错误分类。

## 功能点二：后端列表查询参数支持

### 问了什么

我向AI询问：为了支持前端候选人筛选和分页，后端接口如何接收关键字、岗位、状态、页码和每页数量，并保持Service层职责清晰。

### AI给了什么

AI建议在API层完成参数校验，在Service层组织筛选条件和分页结果，在Repository层执行查询；返回数据应包含列表、总数、页码和每页数量。

### 我实际改了什么

1. 补充候选人列表接口的筛选与分页参数。
2. 在Service层处理默认页码、空筛选条件和返回结构。
3. 前端按照统一字段读取列表和总数。
4. 约定筛选条件为空时不附加对应查询限制。

### 使用反思

AI生成的第一版把部分业务判断直接放进路由层，不符合项目分层约定。我学到接口代码要先明确边界：路由负责接收，Service负责业务，数据访问保持在Repository层。

# 2026年7月10日

## 功能点一：面试安排弹窗与表单校验

### 问了什么

我向AI询问：面试安排页面需要选择候选人、面试官、会议室、日期和时间段，怎样设计表单交互并避免提交不完整或冲突的数据。

### AI给了什么

AI建议对必填项、时间先后关系和面试时长做前端校验；提交前显示确认信息，提交后刷新日历和候选人详情中的面试记录。

### 我实际改了什么

1. 完成面试安排弹窗和基础表单校验。
2. 支持选择候选人、面试官、会议室和预约时间。
3. 未选择必填项或结束时间早于开始时间时禁止提交。
4. 提交成功后更新面试列表，并给出结果提示。
5. 在候选人详情页增加进入排期页面的快捷入口。

### 使用反思

AI处理时间时默认采用浏览器本地格式，曾造成提交格式与后端要求不一致。我认识到日期时间是前后端联调中最容易出错的部分，需要事先约定格式、时区和字段含义。

## 功能点二：面试日历加载状态排查

### 问了什么

我向AI询问：面试日历在请求异常或返回空列表时为什么会一直显示加载中，以及怎样避免切换日期时旧请求覆盖新结果。

### AI给了什么

AI建议在请求的成功、失败和结束阶段都关闭加载状态；日期切换时根据最新查询条件更新结果，并对重复请求做节流或取消处理。

### 我实际改了什么

1. 修正日历请求结束后未关闭加载状态的问题。
2. 空列表时显示“当前日期暂无面试安排”。
3. 请求失败时显示重新加载按钮。
4. 切换日期时重新传递起止时间范围。
5. 避免旧请求返回后覆盖当前日期的数据。

### 使用反思

AI建议的取消请求方案与现有请求封装不完全匹配，不能直接复制。我学到应先确认项目当前技术实现，再选择合适的优化方式，不能把通用方案当成现成代码。

# 2026年7月11日

## 功能点一：管理端响应式样式优化

### 问了什么

我向AI询问：候选人表格、统计卡片、筛选区和面试弹窗在不同屏幕宽度下如何保持可用，尤其是表格列较多时的处理方式。

### AI给了什么

AI建议采用固定断点，在小屏幕下将筛选条件换行、统计卡片改为多行排列，表格使用横向滚动并保留关键列，弹窗宽度根据屏幕调整。

### 我实际改了什么

1. 调整筛选区、统计卡片和按钮组的换行规则。
2. 为候选人表格增加横向滚动容器。
3. 优化弹窗在较小屏幕上的宽度和内部间距。
4. 统一常用断点下的字体和间距变化。
5. 修复部分状态标签换行后显示不完整的问题。

### 使用反思

AI为了兼容小屏幕添加了较多覆盖样式，部分规则反而影响了其他页面。我认识到响应式应优先建立少量稳定断点，避免后期不断叠加临时CSS。

## 功能点二：前端业务状态统一

### 问了什么

我向AI询问：候选人池、详情页、面试日历和报告页面如何统一处理加载、空数据、无权限、请求失败和正常展示五种状态。

### AI给了什么

AI建议每个业务页面都明确区分Loading、Empty、Error、PermissionDenied和Content状态，并提供统一的图标、文字和重试操作。

### 我实际改了什么

1. 为主要招聘页面补充加载、空数据和错误状态。
2. 403时显示无权限说明，而不是展示空白页面。
3. 网络失败时提供重新加载入口。
4. 删除请求失败后自动回退固定列表数据的逻辑。
5. 统一状态文案，避免不同页面对同一问题给出不同提示。

### 使用反思

AI给出的状态组件包含过多装饰性内容，影响业务页面的简洁性。我学到状态提示首先要说明“发生了什么、用户能做什么”，视觉效果应服从信息表达。

# 2026年7月13日

## 功能点一：招聘报告页面与数据展示

### 问了什么

我向AI询问：招聘报告页如何展示岗位数量、候选人阶段分布、来源统计和平均评分，同时保证图表和列表都来自同一份真实数据。

### AI给了什么

AI建议由后端返回统一汇总数据，前端将数据映射为统计卡片、柱状图、饼图和明细列表；没有数据时不渲染误导性的图表。

### 我实际改了什么

1. 设计招聘报告页的统计卡片、阶段分布和来源分布区域。
2. 对接招聘汇总接口，前端不再维护独立的固定统计值。
3. 统一候选人阶段和评分字段的展示口径。
4. 图表无数据时显示说明，避免展示模拟比例。
5. 增加时间范围切换后重新请求报告数据的交互。

### 使用反思

AI生成的图表配置使用了与项目不一致的字段结构，需要根据实际接口重新映射。我认识到图表只是数据的展示，先确认数据契约，再编写图表配置会更稳妥。

## 功能点二：Agent运行面板前端展示

### 问了什么

我向AI询问：多Agent招聘流程如何在前端展示运行状态、节点进度、工具调用摘要和最终报告，同时不暴露内部思维过程。

### AI给了什么

AI建议用运行编号标识一次任务，通过事件流更新节点状态；前端只展示可审计的阶段摘要、时间和结果状态，不展示模型隐藏推理。

### 我实际改了什么

1. 将运行状态、节点列表、事件摘要和最终结果拆分展示。
2. 使用`run_id`关联前端页面与后端运行记录。
3. 展示进行中、完成、待人工复核、跳过和失败等状态。
4. 页面刷新后读取已保存的运行快照恢复展示。
5. 只显示结构化事件摘要，不展示敏感输入和内部推理内容。

### 使用反思

AI曾建议使用固定延迟和模拟日志丰富页面效果，但这会让运行过程失真。我学到Agent页面必须反映真实事件，无法获取事件时应明确提示，而不是制造“正在思考”的假象。

# 2026年7月14日

## 功能点一：前后端字段与状态联调检查

### 问了什么

我向AI询问：候选人列表、详情、评分、面试安排和招聘报告之间有哪些常见字段不一致问题，如何制定联调检查清单。

### AI给了什么

AI建议逐项检查：

1. 候选人编号、申请编号和岗位编号是否混用。
2. 分页字段、评分字段和阶段状态是否一致。
3. 日期时间格式是否统一。
4. 空值、权限不足和不存在数据是否有明确返回。
5. 操作成功后相关列表和详情是否同步刷新。

### 我实际改了什么

1. 检查并修正候选人、申请和岗位标识的使用位置。
2. 对齐评分总分、匹配度、风险标签和流程状态字段。
3. 修正面试时间字段的提交和展示格式。
4. 操作成功后刷新候选人详情、日历和列表中的关联数据。
5. 整理接口异常、字段为空和权限不足的前端表现。

### 使用反思

AI会根据常见命名猜测字段，例如把实际字段改成看似合理但未确认的名称。我认识到联调时必须以接口文档和真实响应为准，AI建议只能作为排查清单，不能直接当作事实。

## 功能点二：代码检查与AI协作复盘

### 问了什么

我向AI询问：在提交验收前，前端和少量后端改动应重点检查哪些问题，才能减少页面可用但业务不正确的情况。

### AI给了什么

AI建议从交互、接口、权限、边界数据和代码结构五方面检查，并提醒关注重复提交、空状态、时间边界、错误提示和分层调用。

### 我实际改了什么

1. 检查主要页面是否存在固定数据、无效入口和错误提示不清晰的问题。
2. 检查筛选分页、详情跳转、面试安排和报告查询的基本流程。
3. 检查后端接口是否仍遵守API、Service和Repository分层。
4. 将发现的问题区分为已修正问题、待人工验收问题和后续优化项。
5. 整理AI辅助开发过程，明确AI负责建议和初版代码，本人负责确认业务规则和最终修改。

### 使用反思

本阶段AI在页面结构、重复代码、排查方向和文案整理上提高了效率；但在真实接口字段、项目既有封装、日期时间和边界条件上仍可能做出不准确假设。后续我会先提供明确的页面需求和接口样例，再让AI处理局部实现，并在提交前通过真实数据完成验证。

# 总结

本阶段以前端工作为主，完成了招聘管理端页面结构、候选人池、详情展示、筛选分页、评分状态、面试安排、招聘报告、响应式适配和Agent运行面板等内容；后端工作主要配合列表筛选分页、数据汇总和接口字段对齐。

AI能够明显减少页面初版、样式整理、重复逻辑和排查清单的编写时间，但不能替代对项目结构、接口契约和业务边界的判断。使用AI时，应先确认真实数据和现有代码约束，再逐步生成、联调和复核，避免将通用示例或猜测字段直接带入项目。
