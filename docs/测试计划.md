## 测试范围

- 单元测试：覆盖 Service、Repository、权限判断和规则计算外层编排。
- API 集成测试：覆盖统一响应、错误返回、身份上下文和权限边界。
- 前端页面状态测试：覆盖加载、空、错误、权限拒绝、Toast、表单和 Agent Trace 展示。
- Agent Tool 调用测试：覆盖 Tool 到 Service 的调用链。
- RAG 来源测试：覆盖制度命中片段和来源展示。
- 权限与脱敏测试：覆盖薪资字段可见范围和拒绝访问。
- 薪资预审状态机测试：覆盖草稿、已生成预审、待 HR 确认、已确认。
- Web 与小程序数据一致性测试：覆盖员工考勤、年假、本人薪资摘要。

## Sprint 2.3 确定性中间版本验收

以下为待人工执行的验收清单，不表示已经运行或通过：

- 使用 `agent.hr.use` 账号创建 Run 并获得真实 `run_id`；无权限账号不能调用接口，不同用户不能读取彼此 Run。
- 执行计划的阶段为 `SPRINT_2_3_DETERMINISTIC_INTERMEDIATE`，下一阶段为 `LLM_RAG_INTEGRATION`；五个执行节点和唯一跳过节点与契约一致。
- 招聘策略、简历解析、岗位匹配、决策审查和 HR 最终报告按真实事件推进；`AGENT_THINKING` 只包含可审计结构化阶段摘要。
- `CandidateEvaluationService` 正确映射人工维护的确定性评分结果，不复制评分算法，不把加分技能缺失算作必备条件失败。
- 人工评分算法不可加载、未实现或返回格式错误时，`overall_score` 与 `job_match_score` 保持空值，`requires_review` 为 `true`，不得生成替代分数。
- 岗位匹配只引用候选人画像中真实存在的证据编号，只沿用 `EnterpriseKnowledgeSummary.sources`，面试问题仅根据真实缺口、低分维度和已有风险确定性生成。
- 面试评估在无真实结构化评价时为 `SKIPPED`，原因为 `STRUCTURED_INTERVIEW_FEEDBACK_NOT_AVAILABLE`，不产生技术、沟通、项目等虚假评价。
- `DecisionReviewService` 不修改岗位匹配分数，`deterministic_score_preserved` 始终为 `true`；缺少面试评价时产生 `INTERVIEW_DATA_MISSING`。
- 可信度按 `0.4 × 证据覆盖 + 0.3 × 画像完整度 + 0.2 × 评分可用性 + 0.1 × 面试可用性` 稳定计算，限制在 0～100 并保留两位小数。
- 岗位匹配或决策审查节点为 `NEEDS_REVIEW` 时，结构化结果正常生成后工作流仍可为 `COMPLETED`。
- `RecruitmentReportService` 对有分候选人按分数降序、同分按编号升序、无分候选人最后排序；来源按 `source_id` 去重，不自行补分或新增来源。
- HR 报告的人才缺口与下一步动作只根据真实结构化结果汇总，`requires_human_decision` 为 `true`，不表示已录用、淘汰、排期或确认薪资，最终决定由 HR 完成。
- SSE 中岗位匹配、决策审查和报告分别使用 `job_match_summary`、`decision_review` 和 `report` summary key，Tool 事件只在真实调用时发布，事件顺序与契约一致。
- Snapshot 包含 `job_matches`、`interview_evaluations`、`decision_reviews` 和 `report`；页面刷新后以 GET Snapshot 为真实来源。
- 前端正确显示岗位匹配、规则审查、结构化报告、`NEEDS_REVIEW` 与面试 `SKIPPED`，不存在假 Tool、假 RAG/ChromaDB、假评分、假面试、随机日志或固定延迟。
- 企业知识明确显示 `LOCAL_HYBRID_FALLBACK`，不得描述为 ChromaDB 命中或真实本地知识库；真实大模型、LangGraph 和 ChromaDB 均未接入。
- Runtime 不访问 Repository 或 `human_only`，数据库 Session 不进入后台 Runner；Agent 与 Tool 不直接导入 `human_only`，Tool 只调用 Service。
- 失败事件只返回安全的 `failed_node`、`failed_step`、异常类型和公共错误说明，不返回异常原文、完整简历或敏感输入。
