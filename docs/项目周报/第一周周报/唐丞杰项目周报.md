## 四、遇到的问题与困难

### 1. 技术问题

- 前端页面涉及七大业务模块，每模块有多个视图和状态变体，早期存在布局风格、字段命名、状态表示和错误提示不统一的问题。
- SSR/SSE 事件流接入时，需要同时处理 Agent 节点状态映射、事件类型解析和前端实时更新，前后端事件格式对齐过程较复杂。
- 共享 API 层需要同时支持 mock 开发模式和后端真实接口切换，且在接口不可用时优雅降级，对错误处理逻辑要求较高。
- 多 Agent 看板需要展示大量实时信息（多个 Agent 节点、多候选人、多 Tool 调用），组件状态管理和渲染性能需要持续关注。
- 响应式布局需要在桌面端、投屏大屏和移动端之间平滑适配，部分复杂组件（如 Agent 流程图、权重滑块）在不同视口下需要不同的布局策略。

### 2. 已采取的解决方案

- 建立 `shared/components/base/` 和 `shared/components/feedback/` 两级共享组件体系，统一所有业务页面的状态展示和交互反馈。
- 通过 `shared/api/modules/*` 统一前端 API 调用入口，所有页面通过一致的接口层获取数据。
- 按照"加载中 → 展示数据 / 空数据 / 错误 / 权限拒绝"的状态机模式重构所有核心页面。
- 对 Agent 事件流建立统一的事件类型枚举和状态映射表，前端按事件类型驱动组件更新。
- 使用 CSS 媒体查询 + Flex/Grid 混合布局，对不同视口宽度采用不同的列数和排列方式。
- 在 `docs/Sprint1-Bug清单.md` 中记录每项 Bug 的复现步骤、优先级、修复方案和回归结果，形成可追溯的问题管理流程。

- 提升了前后端联调过程中的问题定位能力，尤其是 API 代理配置、字段映射、状态同步和错误处理等常见问题的排查方法。

### 2. 对项目和开发流程的思考

前端不仅仅是"把后端数据展示出来"，而是需要构建完整的用户体验闭环：从页面加载到数据展示，从操作反馈到异常处理，每一个状态都需要被精心设计和实现。在多 Agent 系统中，前端更要承担"让复杂的 Agent 协作过程变得可理解"的职责——HR 不需要知道 LangGraph 内部如何运转，但需要看到每个 Agent 在做什么、得出了什么结论、依据是什么。

Bug 清单和回归清单的价值不仅在于记录问题，更在于形成一套可追溯的质量保障流程。每项改动都能对应到具体的 Bug 编号和验收场景，这在多人协作和多 Sprint 迭代中尤为重要。

### 3. 自我改进方面

下周将进一步加强 Agent 异常场景和边界情况的处理，确保 Agent 执行失败、事件流中断和部分数据缺失等情况下前端仍能给出清晰的反馈，不会出现空白面板或无响应状态。同时继续优化多 Agent 看板的展示效果，使系统在答辩投屏时能够清晰传达技术深度和业务价值。
