## 一：项目范围与整体架构确定

### 问了什么

我们向 AI 询问：企业人力资源管理 Agent 项目应如何确定最终范围，如何在较短开发周期内兼顾功能完整度、技术深度和演示效果。

### AI 给了什么

AI建议将项目划分为以下主要模块：

1. 招聘管理
2. 员工服务
3. 考勤管理
4. 薪资预审
5. 权限审计
6. Agent 工作流
7. 企业制度 RAG
8. Web 管理端
9. 微信小程序员工端
10. Gradio 内部调试台

AI还提出使用一套 FastAPI 后端，同时支撑多个前端入口，并采用分层架构：

`API → Service → Repository → PostgreSQL`

Agent功能通过Tool调用业务层：

`Agent → Tool → Service`

### 我们实际改了什么

1. 最终确定项目范围为招聘、员工服务、考勤、薪资预审、权限审计、Agent、RAG、Web和小程序。
2. 确定使用一套FastAPI后端，Web、小程序和Gradio共用后端接口。
3. 普通业务请求采用API、Service、Repository和PostgreSQL分层。
4. Agent不直接访问数据库，而是通过Tool调用Service。
5. 将Gradio定位为内部调试工具，不作为正式业务入口。
6. 根据项目周期删除过于庞大、短期无法完成的功能。

5. 后端统一使用FastAPI，前端统一使用Vue。

## 功能点三：薪资访问控制代码理解

### 问了什么

我们向AI询问：`salary_access_control.py`为什么会返回不同格式，字典参数分支和角色权限分支分别在做什么。

### AI 给了什么

AI解释：

1. 参数为角色字符串时，函数进行字段级薪资访问判断。
2. 本人权限可读取本人完整字段。
3. 全量权限可读取完整字段。
4. 部门管理权限只能读取有限字段。
5. 脱敏权限只能读取部分字段。
6. 参数为字典时，函数返回薪资预审摘要。

### 我们实际改了什么

1. 明确薪资访问判断和薪资预审是两种不同业务。
2. 检查本人、全量、部门和脱敏权限顺序。
3. 保留无权限时默认拒绝。
4. 要求所有敏感访问写入审计日志。
5. 识别同一函数返回两类结构不利于维护的问题。
6. 后续计划将薪资访问判断和薪资预审拆分为两个函数。

4. 排期确认前继续检查候选人、面试官和会议室是否重复占用。
5. 在共同时间窗口中按照15分钟步长生成固定时长候选方案。
6. 起始方案冲突时继续搜索后续时间。
7. 只有整个共同窗口都无法安排时才返回无可用时间。
8. 补充对应的算法和Service测试场景。
9. 同步更新排期算法说明和接口契约。

### 使用反思

AI帮助定位了“冲突记录”和“最终推荐冲突”被混用的问题，但真正修复需要同时修改算法、Service响应结构和前端判断逻辑。跨层字段含义如果没有统一，即使每一层代码单独看起来正确，整体功能仍然可能失败。

## 功能点三：六节点LangGraph架构深化

### 问了什么

我们向AI询问：7月16日的LangGraph只是在Graph节点中调用原有完整Pipeline，如何进一步让招聘策略、简历解析、岗位匹配、面试评估、决策审查和HR报告成为真正独立的LangGraph节点。

### AI给了什么

AI建议：

1. 将六个业务阶段分别实现为可执行Graph Node。
2. Graph State只保留`run_id`、执行入口和人工审核路由标记。
3. Store、Tool和模型依赖通过运行时注入。
4. 完整流程从招聘策略依次执行到决策审查。
5. 决策审查发现待人工审核内容时结束本次Graph执行。
6. 人工审核全部完成后，从报告入口继续执行HR报告节点。
7. 没有真实面试评价时，面试评估节点仍需进入并记录`SKIPPED`。
8. Agent Run Store继续作为唯一持久化状态来源。
9. 不使用Graph Checkpointer保存第二份业务状态。

### 我们实际改了什么

1. 将六个招聘阶段改为独立LangGraph节点。
2. 建立完整工作流入口和仅生成HR报告的续接入口。
3. 招聘策略、简历解析、岗位匹配、面试评估和决策审查按顺序执行。
4. 决策审查后根据是否需要人工复核进行条件路由。
5. 无待复核内容时直接进入HR报告。
6. 存在待复核内容时结束本次Graph执行并等待人工处理。
7. 审核完成后通过同一Graph的报告入口生成最终报告。
8. 保留原有SSE、Snapshot、Run持久化和幂等规则。
9. 面试评价数据不存在时仍记录节点执行，并将状态标记为`SKIPPED`。
10. 更新系统架构说明和相关架构决策记录。

### 使用反思

AI提供的Graph设计能够快速形成可运行架构，但团队需要识别“使用了LangGraph”和“每个Agent都是真实独立节点”之间的区别。第一次接入只是把旧流程包装进Graph，第二次重构才真正提高了编排深度和可解释性。

## 功能点四：最终演示界面与项目材料收尾

### 问了什么

我们向AI询问：答辩前如何调整首页、招聘报告、多Agent看板和面试评估节点，使页面既有较好的展示效果，又不会将尚未具备真实数据的功能展示为已经完成。

### AI给了什么

AI建议：

1. 统一首页卡片、图表和页面间距。
2. 删除与当前页面功能重复的全局输入栏。
3. 多Agent看板按照实际执行顺序排列。
4. 未接入真实面试评价时，将面试评估显示为待完成或跳过。
5. 不允许点击没有处理依据的节点。
6. 数据量不足时可以显示明确的演示数据，但不能伪装成实时数据库结果。
7. 同步修改README、接口文档、PPT和项目总结。
8. 文档必须区分已完成、待人工验收和计划中功能。

### 我们实际改了什么

1. 调整Web首页和招聘看板的展示效果。
2. 优化招聘热力图、招聘报告图表和部门数据展示。
3. 删除页面底部重复的全局Agent输入栏。
4. 调整多Agent节点顺序和页面布局。
5. 将没有真实面试评价数据的节点显示为“面试评估，待完成”。
6. 禁止打开没有实际处理依据的面试评估详情。
7. 优化简历和多Agent流程的显示问题。
8. 修正部分浅色、深色和悬浮状态样式。
9. 更新项目README、总结报告、周报和PPT材料。
10. 对代码存在但尚未完成真实验收的功能保留明确说明。

### 使用反思

AI可以帮助快速生成展示方案，但演示数据、真实数据和计划中功能必须明确区分。为了页面好看而使用固定数据时，应将其定位为演示回退，不能在答辩中描述为数据库实时计算结果。最终展示效果和功能真实性需要同时保证。
